# Findings

- [major] **Sibling resolver entry point `resolve_file_target_to_open_in_warp` still routes images to `CodeEditor`.**
  `app/src/util/openable_file_type.rs:167-181` is a second resolver entry point that is NOT short-circuited. It is reached from two production call sites:
    - `app/src/code/file_tree/view.rs:2190-2199` — when `editor_layout.is_some()` (e.g. "Open in New Tab" with an explicit layout). This is the file-tree path that 1b is supposed to fix per tech.md §74 / §519 ("file_tree/view.rs:2202-2208"). A user who right-clicks `screenshot.png` from the file tree and picks "Open in New Tab" / "Open in Split Pane" today still lands in `FileTarget::CodeEditor(...)`, not `FileTarget::ImagePreview`. SVG additionally fails as raw XML in the editor.
    - `app/src/uri/mod.rs:1078` — `OpenFileAction::Editor` URI flow.
  Tech.md §74 phrasing ("the resolver" / "step 0 ahead of step 1") could be read as targeting only `resolve_file_target_with_editor_choice`, but §519's acceptance criterion ("a click on `screenshot.png` produces a `CodePanelsFileOpened` event with `target = "image_preview"`") cannot be satisfied via the explicit-layout branch in `file_tree/view.rs` without a matching short-circuit in `resolve_file_target_to_open_in_warp`. Recommend either (a) adding the same `if is_supported_image_file(path) { return FileTarget::ImagePreview; }` at the top of `resolve_file_target_to_open_in_warp`, or (b) routing the explicit-layout case through `resolve_file_target_with_editor_choice` for image files.

- [major] **Two `is_supported_image_file → SystemGeneric` bypasses still live ahead of the resolver.** The commit removes the bypass in `notebooks/link.rs`, but two structurally identical bypasses remain:
    - `app/src/ai/ai_document_view.rs:919-923` — `if is_supported_image_file(path) { FileTarget::SystemGeneric } else { resolve_file_target(...) }`. AI document view "open file" path.
    - `app/src/ai/blocklist/block.rs:273-275` — `detected_file_path_target_override` returns `Some(FileTarget::SystemGeneric)` for images, which is consumed ahead of the resolver in the AI-block detected-link flow.
  These are flagged in the prompt as candidates for removal. The 1b commit message says "The resolver is the single source of truth now" but it is not yet — these two sites continue to pre-empt it. Whether they belong in 1b or a follow-up bullet is a scoping call (the IMPLEMENTATION_TODO.md does not list them explicitly), but the commit message is factually inaccurate as it stands. Either tighten the message, or include the removals here.

- [minor] **Telemetry acceptance criterion not yet reachable from the file-tree path.** Because of the first finding, the §519 acceptance criterion ("a click on `screenshot.png` ... `target = "image_preview"`") only fires when `editor_layout.is_none()`. The 7-bullet telemetry test (commit `ec6ffb1`) verifies the serialization string, but the end-to-end "click PNG in tree → telemetry says image_preview" guarantee is conditional on user state. Worth a note in either the 1b commit or the 7 review.

- [nit] Step-0 placement is correct: short-circuit lands ahead of `is_file_openable_in_warp` (markdown/binary classification), the `EditorChoice::Warp` arm, the `EditorChoice::EnvEditor` arm, and the binary fall-through to `SystemGeneric`. Verified against §74 lines 96-114 and the file at lines 207-247.

- [nit] Predicate purity (`is_supported_image_file`) is extension-only and case-insensitive — matches §67's "Reused as-is." No content sniffing, no FS access, no metadata calls. The actual regular-file / size gating lives in the workspace 2-arm (commit `38aec6e`), which is the documented layering. An attacker-controlled `evil.png` whose contents are HTML/PE flows through the resolver as `ImagePreview`, then hits the 2-arm `metadata` size cap and the 64 MB asset-cache cap and the decoder caps (4a/4b/4c) before any pixels ever reach the renderer. This is the intended defense-in-depth and is correct. The predicate does not widen the attack surface beyond what the decode pipeline already bounds.

- [nit] Symlink / non-regular-file behavior: resolver does no FS check, so a symlink to a 200 GB sparse file with extension `.png` reaches `ImagePreview` and is rejected by the 2-arm `metadata`/`is_file()` check. Correct layering.

- [nit] `feature = "local_fs"` off: `is_supported_image_file` is NOT cfg-gated (def at lines 76-87), but `resolve_file_target_with_editor_choice` IS (`#[cfg(feature = "local_fs")]` at line 199). The short-circuit therefore only runs in `local_fs` builds, which is the only place ImagePreview wiring exists end-to-end. Correct.

- [nit] `notebooks/link.rs` removal is safe in isolation: the new `target` value flows into `LinkEvent::OpenFileWithTarget`, whose handler is the same workspace `open_file_with_target` that has the `ImagePreview` arm wired. No test fixtures depend on the old `SystemGeneric` shape — `git log -p app/src/notebooks/link.rs` shows the bypass was added without an accompanying test, and there are no tests in the repo that assert `LinkEvent::OpenFileWithTarget { target: FileTarget::SystemGeneric }` for images.
