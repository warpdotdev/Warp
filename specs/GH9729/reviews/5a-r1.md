## TOCTOU windows (per the lens checklist)

Walked all five windows. The defenses compose correctly:

1. Workspace pre-stat → asset-cache `open(O_NONBLOCK)`. Path-swap to FIFO between the two: `open()` returns immediately rather than hanging because `O_RDONLY | O_NONBLOCK` on a FIFO with no writer is documented to return immediately (POSIX `open(2)`). The post-open `is_file()` then bails. Pass.
2. fstat-via-fd → read. File-size growth caught by `MAX + 1` bounded read. Pass.
3. Peek → take/read_to_end share the same fd cursor. Verified `async_fs::File`'s read advances the position; the buffered peek + `take` continuation does not double-count and does not re-read the prefix. Pass.
4. Peek says non-SVG → 64 MB cap, attacker overwrites with SVG. Read up to 64 MB; SVG branch downstream is protected by the §300 dim cap (4a/4b). Acceptable per the layered defense. Pass.
5. Peek says SVG → 4 MB cap, attacker overwrites with raster. Truncated raster fails to decode and surfaces as `FailedToLoad` → user-facing `"could not decode image"`. Acceptable. Pass.

## `O_NONBLOCK` semantics

Verified per Linux/macOS man pages: `O_NONBLOCK` has no effect on `read(2)` of a regular file (reads always block until disk I/O completes). It changes `open()` semantics for FIFOs (returns immediately for `O_RDONLY` regardless of writer presence) and changes `read()` to return `EAGAIN` on FIFOs/sockets/character devices when no data is ready. Since the post-open `is_file()` rejects all non-regular targets before any read, the only path that reaches `read` is a regular file, where `O_NONBLOCK` is a no-op. The spec's claim that no `fcntl` clear is needed is correct. Pass.
