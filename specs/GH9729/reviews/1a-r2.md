# Findings

- [nit] Placement of the no-op arm in `app/src/workspace/view.rs` is at the **end** of the `match target` block (after `SystemGeneric`), not "next to `MarkdownViewer`" as `tech.md` §74 (line 64) prescribes. The spec is explicit: "The new arm slots in next to `MarkdownViewer`." Item 2b will fill in the body, but the structural placement is established by 1a and a later refactor would have to move it. Sibling arms are roughly grouped Warp-internal first (MarkdownViewer, EnvEditor, CodeEditor) then external/system (ExternalEditor, SystemDefault, SystemGeneric); placing `ImagePreview` (Warp-internal Lightbox) at the very end after `SystemGeneric` weakens that grouping.
- [nit] `FileTarget::ImagePreview` doc-comment uses a 3-line block (summary + blank line + explanation about no payload), whereas every sibling variant on the enum uses a single-line `///` comment (`MarkdownViewer`, `CodeEditor`, `ExternalEditor`, `EnvEditor`, `SystemDefault`, `SystemGeneric`). The extra context is genuinely useful, but it is stylistically inconsistent with the file's existing convention. Either trim to one line or leave a tracking note that the longer style is intentional.
- [nit] The view.rs no-op arm body is a 5-line comment with empty braces, where `FileTarget::ImagePreview => {}` (single line, with the comment moved one line above the arm) would be more idiomatic and would not draw a clippy hit on `clippy::no_effect_underscore_binding` style nits in some configurations. Multi-line is defensible because it documents two cross-references (TODO 1b + TODO 2b), so this is borderline; the version chosen is acceptable.
- [nit] The view.rs comment uses an em-dash ("see TODO 1b — so this arm…"). Not a Rust concern, but the codebase elsewhere reads as plain ASCII; an en-dash or a comma would be lower-friction and easier to grep.
- [minor] The view.rs comment says the arm is "unreachable in practice"; the more idiomatic Rust expression of "we know this arm cannot fire yet" would be `unreachable!("…")` rather than a silent no-op, which would surface the contract at runtime if 1b is ever shipped before 2b. The author chose silent no-op deliberately (commit message: "until item 1b adds the resolver short-circuit, no caller can produce this variant"), so this is a judgement call; documenting the rationale for **not** using `unreachable!` would help future readers.
