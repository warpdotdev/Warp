# Findings

- **`set_limits` survives `into_frames()` for both decoders.** Verified against `image-0.25.9`: `GifDecoder::into_frames(self)` (`codecs/gif.rs:425`) consumes the decoder and threads it into `GifFrameIterator::new(self)`, so the `self.limits` field set by `set_limits` (`codecs/gif.rs:100`) is carried by the iterator and applied per frame as it reads. `Frames<'a>` (`animation.rs:8-16`) is just a thin `Box<dyn Iterator<Item = ImageResult<Frame>> + 'a>` wrapper, so its lifetime is the lifetime of whatever the impl-decoder borrows; in this code each `Cursor::new(data)` borrows `data: &[u8]`, so the iterator's `'a` is the function-arg lifetime and there is no use-after-set hazard. The two-line `let mut dec = ...; dec.set_limits(...)?; dec.into_frames()` pattern is correct, and the limits are durable. Worth a one-line comment in the helper acknowledging this — the question "do limits survive `into_frames`" is the first thing a reviewer reaches for, and the spec already calls it out at §259.

- **Match on `image::ImageFormat` with a `_ => bail!`. Acceptable, but `unreachable!()` would have been more honest.** The function is private (`fn`, not `pub fn`) and called from exactly two sites in this same file, both of which pass a literal `ImageFormat::Gif` or `ImageFormat::WebP`. The catch-all is genuinely a programmer-error path — there is no caller-supplied format to defend against. Spec §259 itself uses `_ => unreachable!()` (line ~280). The implementation diverges to `anyhow::bail!("decode_animated_with_limits called with non-animated format")`, which is defensible (no panic on degenerate input) but tonally inconsistent with the other three error strings, which describe input pathology, not programmer error. Two clean options: (a) follow the spec literally with `unreachable!("decode_animated_with_limits called with non-animated format")`, which encodes the invariant in the type system's eyes and removes a never-hit branch from the surface area, or (b) tighten the parameter to an inner `enum AnimatedFormat { Gif, WebP }` so the match is exhaustive without an arm. (b) is the cleanest in isolation but breaks the symmetric `(data, ImageFormat)` signature shared with `decode_static_with_limits`, so I'd take (a).

- **Trait-object alternative is not worth it.** The `match { Gif => GifDecoder::new(..); WebP => WebPDecoder::new(..) }` returning `Frames<'_>` looks like duplication, but the duplication is exactly two lines per arm (`let mut dec = ...::new(...)?; dec.set_limits(decode_limits())?;`) and the type that unifies them is already `Frames<'_>` itself — `into_frames(self)` is the unifying call. A `Box<dyn AnimationDecoder>` rewrite would have to also unify `set_limits`, which is on `ImageDecoder` not `AnimationDecoder`, so you'd end up with a trait-object that satisfies both bounds — strictly more code than what's there. Keep as-is.

- **Iteration shape is idiomatic and correct.** `for (i, frame) in frame_iter.enumerate()` with `if i >= MAX_ANIMATED_FRAMES { bail }` *before* the `frame?` unpack is the right order: we refuse the over-cap input without paying for the (n+1)th frame's decode work. `let frame = frame?;` shadow is idiomatic. `frame.buffer().width()/height()` cast `u32 → u64` then `saturating_mul`/`saturating_add` is the right defensive arithmetic for an 8192² × 256-frame budget that fits comfortably in `u64`. `frames.is_empty()` post-iteration is the right defensive guard against a truncated input that yields zero frames without an iterator-level error. Nothing to change.

- **`drop(decoder)` in the animated-WebP branch carries its weight.** Same analysis as 4a-r2 found for the static-WebP branch: `WebPDecoder<R>` consumes `R = Cursor<&[u8]>` which holds an immutable borrow, so a second `Cursor::new(data)` inside the helper coexists with the original — the explicit `drop(decoder)` is documentation, not borrow-checker-required. I'd keep it for the same reason 4a's was kept: the comment ("GH9729 §259: bound frame count and total pixels.") sits right above and is the visible seam between the cheap `has_animation()` probe and the limits-aware re-decode. Without the `drop`, the seam vanishes and the next reader has to mentally trace which `Cursor` is alive. Keep.

- **Comment-to-code ratio is calibrated.** The two new constants get 4-line doc-comments, each of which (i) states the cap, (ii) explains why the standard `image::Limits` envelope is insufficient (decoder-honesty asymmetry), and (iii) cites `tech.md §259`. The function gets an 8-line doc-comment that re-states the per-decoder weakness rationale. There is some repetition between the constants' "decoders do not enforce frame counts via `image::Limits`" and the function's "`GifDecoder` ignores `max_alloc` per frame; `WebPDecoder` does not override `set_limits` at all", but the two phrasings target different readers (the const docs explain the cap; the function doc explains why the cap is the right tool). Proportionate to 4a's `decode_limits()` + `decode_static_with_limits()` doc weight. Keep.

- **Function placement: `decode_limits → decode_animated_with_limits → decode_static_with_limits` is a defensible ordering, but not the obvious one.** The new helper is sandwiched between `decode_limits` (which it consumes) and `decode_static_with_limits` (which is sibling, not parent). Three ordering principles are at play: (a) alphabetical by name (`animated` < `static`), (b) call-graph dependency (`decode_animated_with_limits` calls `decode_limits()`, so it should follow it; same for static), (c) section pairing (consts → static helper → consts → animated helper). The diff picks (a). I'd weakly prefer (c) — group each cap with its consumer — because the static-section reader currently has to skip past the animated helper to reach `decode_static_with_limits`, and vice versa. Cosmetic, not blocking; mention to author and let them pick.

- **Error string tone — three of four are consistent.** `"animated image has too many frames"`, `"animated image exceeds total pixel budget"`, and `"animated image has no frames"` all follow the same shape: subject + verb-phrase describing the input pathology, no terminal punctuation, no source-location detail. The fourth, `"decode_animated_with_limits called with non-animated format"`, names the internal function and reads as a programmer-error message — this is the inconsistency that makes the case for `unreachable!()` (above). The user-visible static-decode strings (`"image is too large to preview"`) are tonally similar — terse, no jargon, no internal symbols — so this string sticks out. Same fix as the match-arm bullet.

- **Tests live in `57be862` and look thorough.** Confirmed via `git show --stat`: the 4-tests-b commit adds 4 tests in `image_cache_tests.rs` (`decode_animated_rejects_too_many_frames`, `decode_animated_rejects_total_pixel_budget`, `decode_animated_constructs_bitmap_for_legitimate_input`, `decode_animated_constructs_bitmap_for_legitimate_webp`) covering both rejection paths and both happy-path codecs. The split into `decode_animated_with_limits_inner` (parametrized) and the production-cap wrapper is a clean refactor for testability — small caps + existing fixtures avoid having to synthesize a 257-frame pathological GIF at test time. Tests are appropriate scope for 4b's risk surface and are not blocking on the 4b commit itself. (Note: I'm reviewing `a6aaf00` per the prompt; `57be862` is the test commit and is already merged into the branch state.)

- **Consistency with 4a is good.** Same `anyhow::Result` return type, same private-by-default visibility, same `image::Limits` envelope reused, same `Cursor::new(data)` shape, same trait-method-import discipline (`use image::ImageDecoder` added once because `set_limits` is a trait method on `GifDecoder`/`WebPDecoder`). Symmetric `decode_static_with_limits` / `decode_animated_with_limits` pair reads cleanly at call sites.

# Suggestions

- **(nit, recommended)** Replace the catch-all `_ => anyhow::bail!("decode_animated_with_limits called with non-animated format")` with `_ => unreachable!("decode_animated_with_limits called with non-animated format")`. Matches `tech.md §259` literally, removes a never-hit error string from the surface, and ditches the one tonally-inconsistent message. One-line change.
- **(nit, optional)** Add a single-line comment after `dec.set_limits(decode_limits())?;` in each arm: `// Limits are carried into the frame iterator: GifFrameIterator owns the decoder.` Saves the next reviewer the round trip into the `image` crate that R2 just made.
- **(nit, optional)** Consider re-grouping helpers as `decode_limits → MAX_ANIMATED_* → decode_animated_with_limits → decode_static_with_limits`, or moving the `MAX_ANIMATED_*` constants down adjacent to their helper. The current `MAX_ANIMATED_*` block sits among the static consts, then `decode_limits` (static), then `decode_animated_with_limits` (animated), then `decode_static_with_limits` (static again) — readable but slightly zigzags. Cosmetic.
- **(not blocking)** No change needed to the test split — `_inner` parametrization in `57be862` is the right call. The refactor pattern (production-wrapper + parametrized inner) is worth lifting to a comment in `tech.md` if it becomes the convention for the rest of the GH9729 helpers; right now both static and animated helpers use it.
