For a v1 with two call sites, the duplicated-but-cross-referenced constant is acceptable. If a third surface ever needs the same cap, the right move is to lift it. Nit-worthy, not blocking.

- **Constant placement at end of file**: the two constants and the helper are appended after the `impl SingletonEntity for AssetCache {}` block. - **Inline `use` statements inside the body**: `use futures_lite::AsyncReadExt as _;` and `#[cfg(unix)] use async_fs::unix::OpenOptionsExt as _;` are inside the function body. The `cfg(unix)` use *should* stay inside (or be cfg-gated at module scope) to avoid unused-import warnings on Windows. The `AsyncReadExt as _` use could hoist to module scope, but keeping it co-located with the only consumer makes the helper self-contained and is a reasonable style choice for a single-use helper. Both `as _` aliases are correct since only the trait methods are needed. Style-only.
- **Block-scoped opts builder**: `let mut file = { let mut opts = ...; ... ; opts.open(&path).await? };` is a clean way to scope `opts` to construction. Reads well; `cargo fmt` keeps it tidy.

The mild inconsistency in wording is a feature, not a defect.
- **Defensive arithmetic**: `(cap + 1).saturating_sub(buf.len() as u64)` ŌĆö `cap + 1` cannot overflow (`cap` is at most 64 MB), `saturating_sub` is just-in-case for when `n > cap + 1` (which should be impossible since `n Ōēż 1024` and `cap Ōēź 4 MB`). Belt-and-suspenders, not wrong.
- **Final `if buf.len() as u64 > cap { bail }`**: this is the deterministic over-cap detector. With `take(cap + 1 - n)` the upper bound on `buf.len()` is `cap + 1`, so the check fires iff the underlying file had more than `cap` bytes. Correct.
