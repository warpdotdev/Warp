# Findings

- **Builder shape for `image::Limits` is the only available idiom in 0.25.9.** R2 prompt asks whether `Limits { max_image_width: ..., max_image_height: ..., max_alloc: ..., ..Default::default() }` would be more idiomatic. It would not — the workspace is pinned to `image = "0.25.9"` (`Cargo.toml:160`), and `image::io::Limits` is `#[non_exhaustive]` with public fields in that version (`~/.cargo/registry/src/.../image-0.25.9/src/io/limits.rs:32-33`). `#[non_exhaustive]` forbids struct-literal construction outside the defining crate even with a `..Default::default()` rest, so the `Default::default()` + `Some(...)` field-assignment shape the commit uses is in fact the only public-API-stable form. The implementation matches the spec verbatim (`tech.md:237-243`). No change requested. Worth recording explicitly because future image-rs upgrades that drop `#[non_exhaustive]` (unlikely) could open up the literal form, but until then this is correct.

- **Helper-function placement is appropriate.** `decode_limits` and `decode_static_with_limits` sit at module scope, immediately above `impl Asset for ImageType` (lines 341-349, 454-460 in the post-abab042 file; the 926dedf snapshot put them slightly higher but in the same band). Module scope is the right home: they are pure data-in/data-out helpers, they are reused by `decode_static_with_limits_inner` for tests after abab042, and the file already follows the "constants + free helpers + impl block" layout that `MIN_REFRESH_DELAY_MS` (line 30) and the SVG_FONT_DB plumbing (lines 32-40) established. Putting them inside `impl Asset for ImageType` would have buried them under a single-method impl and complicated the later test-only `_inner` split. Right call.

- **`drop(decoder)` is not borrow-checker-required, and earns its keep as documentation.** I verified that `WebPDecoder<R>` consumes `R = Cursor<&[u8]>` (`image-0.25.9/codecs/webp/decoder.rs:19-21`), and the inner cursor only holds an immutable borrow of `data`. A second `Cursor::new(data)` inside `decode_static_with_limits` borrows `data` again, also immutably, so the two cursors coexist without `drop(decoder)`. The explicit drop is therefore documentation, not necessity. I think it is **worth keeping** here for two reasons: (1) the comment block immediately below it explicitly calls out "the decoder is dropped here and re-opened by `ImageReader::with_format` inside the helper; the cost is one extra header parse" — the `drop` is the load-bearing referent of "dropped here", and removing it makes the comment dangle; (2) this is the seam between the cheap `has_animation()` probe (which needs the `WebPDecoder` directly) and the limits-aware re-decode (which goes through `ImageReader`). Marking the seam visibly is helpful for any future reader who would otherwise wonder "wait, why don't we just use `decoder` for the static branch too?". Keep as-is.

- **Comment density is proportionate to the security weight on the consts; one match-arm comment is restating the obvious.** The three `MAX_DECODE_*` consts each get a 3-paragraph doc block with a `tech.md` xref (lines 271-286). Across the file they sit alongside three more security-weighted consts (`MAX_ANIMATED_FRAMES`, `MAX_ANIMATED_TOTAL_PIXELS`, `MAX_SVG_RENDER_DIMENSION`, lines 288-304) that adopt the same density, so 4a establishes a pattern that 4b/4c then follow consistently. For security-budget constants this density is appropriate: every cap encodes a magic number that the reader will want to understand without leaving the file. Conversely, the two `// GH9729 §234: enforce dimension/alloc/pixel envelope.` arm-comments at lines 529 and 536 are restating what the helper name already says (`decode_static_with_limits`) and the cross-ref already conveys at the helper definition. One xref at the top of the `match` would have done the job. Minor — not worth a follow-up commit, but worth the style note.

- **Naming `decode_static_with_limits` is fine but slightly mismatched against `decode_animated_with_limits`.** The `_with_limits` suffix scans well in isolation. After 4b lands `decode_animated_with_limits` (verified, line 413 in post-4b state), the pair is symmetric — both helpers are named "decode <kind> with limits", which is a plausible convention. The alternative `decode_static_within_envelope` from the prompt would have echoed the spec's "size envelope" language but broken the symmetry with the animated helper. Keep.

- **Format pass-through is the right call.** The R2 prompt asks whether `decode_static_with_limits(data, format)` could derive `format` internally via `image::guess_format(data)`. It could, but the caller already disambiguated via `image::guess_format` in the outer match at line 527, and re-running `guess_format` inside the helper would (a) re-parse the header for no benefit, (b) push the format-resolution failure mode into the helper and obscure that the helper is only ever called from arms where `format` is already known. Pass-through is the right factoring.

- **`anyhow::Result` at this layer is acceptable.** A typed error would be nicer in principle (`enum DecodeError { Limits, Pixels, Underlying(image::ImageError) }`), but `try_from_bytes` already returns `anyhow::Result<ImageType>`, the existing decoder error path was already `anyhow::Result` via `?`-conversion of `image::ImageError`, and the post-load callback in `d5fdacc` ("post-load callback rewrites failures to Error variant") rewrites all decode failures to a single sanitized `LightboxImageSource::Error { message }` with no message-content branching by error kind. So a typed error would deliver no concrete win at the consumer side. Acceptable.

- **`anyhow::bail!("image is too large to preview")` matches §119 sanitized-message pattern.** No path, no width/height, no underlying error string. Consistent with the SVG branch's `"svg dimensions exceed render budget"` wording (line 481) and with the §182 `Error` variant's "do not interpolate raw inputs" invariant from 3a. Good.

- **Saturating arithmetic on the post-decode pixel count is correctly applied.** `(img.width() as u64).saturating_mul(img.height() as u64)` — the cast widens before multiplication (so `u32::MAX * u32::MAX` does not wrap before the saturate), and `saturating_mul` then caps at `u64::MAX`, which is well above `MAX_DECODE_PIXELS = 67M`, so the comparison is total. Belt-and-suspenders without UB risk. Good.

- **No tests in 4a is the right split.** Confirmed by `git show abab042 --stat`: 4-tests-a (commit `abab042`) adds the unit-test suite — `decode_static_rejects_dimensions_over_cap`, `decode_static_rejects_pixels_over_cap`, `decode_static_accepts_normal_photo`, plus six SVG cases — and refactors `decode_static_with_limits` into a thin wrapper over `decode_static_with_limits_inner(data, format, limits, max_pixels)` so the tests can pass small caps against modest fixtures (the commit message explicitly calls this out: "tests pass small limits and small max_pixels values to exercise rejection paths against modest fixtures"). The 4a commit standing tests-free is therefore a clean split: 4a establishes the production-constant code path, 4-tests-a opens it up for testing via the inner refactor. Acceptable.

- **One forward-looking nit: the wrapper/inner refactor would have read better if introduced in 4a.** This is a 4-tests-a finding, not a 4a one, but worth flagging here because it speaks to whether 4a "ships ready". Splitting into `decode_static_with_limits` (production) and `decode_static_with_limits_inner` (test-parameterized) at 4a-time would have meant the test commit only adds tests, not test-and-refactor. As-shipped, abab042 carries both an additive test suite and a non-trivial signature change to a function that was 8 commits old. Minor — not blocking 4a — but a process note.

- **Module structure is canonical per the spec and consistent with siblings.** §234 prescribes `crates/warpui_core/src/image_cache.rs` and the helper signatures verbatim. The commit places the helpers exactly there. ✓

- **Cargo version pin is healthy for this commit.** `image = "0.25.9"` is the version against which §234 was written ("`ImageReader::limits` is the documented bound for `decode()` in `image` 0.25.x", `tech.md:257`). No silent upgrade risk during this PR's lifetime since the workspace `Cargo.toml` pins to a specific minor.

Either replace them with a single comment above the `match` (e.g. `// Static-raster arms route through decode_static_with_limits for the GH9729 envelope. Animated arms hand off to decode_animated_with_limits below.`) or drop them entirely. Optional, cosmetic.

- **Keep `drop(decoder)` and the comment block** in the static-WebP branch. They correctly mark the seam between probe and re-decode, and the comment specifically names the cost ("one extra header parse"), which is exactly the kind of perf-hit-with-a-reason annotation that pays back the next time someone profiles this path. No change.

- **For 4-tests-a reviewers (not 4a):** flag that the inner/outer refactor was deferred from 4a into the test commit, mixing a non-test change into a test-only commit. Not a 4a finding.

- **Optional documentation polish:** the `MAX_DECODE_PIXELS` doc comment (lines 277-279) could mention that the post-decode check is total because decode time is bounded once the per-axis limits in `image::Limits` have already gated the input — the reader currently has to triangulate that from §234. One extra sentence. Cosmetic.

- **No further changes required to ship 4a.** The implementation is spec-faithful, the helpers are placed correctly, the const choices are internally consistent (`8192² × 4 = 256 MB = MAX_DECODE_ALLOC`), the commit is appropriately scoped, and the deferred test split is clean.
