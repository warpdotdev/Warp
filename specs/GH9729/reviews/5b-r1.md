  `post_load_callback_rewrites_unrecognized_to_error`. ✓

- The §182 sanitize contract: "collapses to a small set of categorical

  too large to preview`) and never interpolates the raw error or any

## Substring map — TWO bail strings introduced in this PR are mis-categorized

`sanitize_load_error` lower-cases the rendered error and matches in
order: `"too large" | "exceeds"` → too-large; `"decode" | "format"` →
decode; else → read.

Walking every bail string introduced in commits 4a/4b/4c/5a:

| Bail string | Substring hit | Mapped category | Correct? |
|---|---|---|---|
| `image is too large to preview` (4a, image_cache.rs:442) | `too large` | too-large | ✓ |
| `animated image has too many frames` (4b, :385) | none | **could not read image** | **WRONG — should be too-large** |
| `animated image exceeds total pixel budget` (4b, :392) | `exceeds` | too-large | ✓ |
| `animated image has no frames` (4b, :398) | none | could not read image | acceptable |
| `decode_animated_with_limits called with non-animated format` (4b, :379) | `decode`, `format` | could not decode | acceptable for unreachable internal-mismatch path |
| `svg dimensions exceed render budget` (4c, :481, :485) | none — source is "exceed render", `contains("exceeds")` returns FALSE | **could not read image** | **WRONG — should be too-large** |
| `local asset is not a regular file` (5a, asset_cache.rs:537) | none | could not read image | acceptable, generic |
| `local asset exceeds size cap` (5a, :554) | `exceeds` | too-large | ✓ |

Verified by hand:
- `"animated image has too many frames".to_lowercase().contains("too large")` → false; `.contains("exceeds")` → false. Falls through.
- `"svg dimensions exceed render budget".to_lowercase().contains("exceeds")` → false (`"exceed render"` ≠ `"exceeds"`). `.contains("decode")` → false. `.contains("format")` → false. Falls through.

These exactly re-confirm:
- the **4b R1 finding** ("too many frames" doesn't match) — still present.
- the **4c R2 finding** (SVG `exceed` vs `exceeds`) — still present.

Both surfaces would, at the user-facing layer, mis-label what is a
"too large to preview" condition as a generic "could not read image."
The user is told to retry rather than that the file is too big — the
exact UX confusion §182 is supposed to prevent.

## image-rs Limit error — also mis-categorized

`image::Limits` violations bubble up via `ImageError::Limits` →
`LimitError::fmt` (image-0.25.9 src/error.rs:517-528):

| Variant | Display string | Substring hit | Mapped category | Correct? |
|---|---|---|---|---|
| `LimitErrorKind::DimensionError` | `Image size exceeds limit` | `exceeds` | too-large | ✓ |
| `LimitErrorKind::InsufficientMemory` | `Memory limit exceeded` | none — `"exceeded"` ≠ `"exceeds"` | **could not read image** | **WRONG — should be too-large** |
| `LimitErrorKind::Unsupported` | `The following strict limits ...` | none | could not read image | acceptable |

Since 4a's `decode_static_with_limits` deliberately sets
`Limits::max_alloc` (the binding constraint per §219-§221) and the
limit-trip path is the *primary* expected too-large surface for the
PNG/JPEG bomb-class attacks the spec calls out, this is a load-bearing
mis-categorization. It is the third independent miss caused by the
same root cause — substring matching against `"exceeds"` does not
catch every reasonable phrasing of "exceeded".

`ImageError::Decoding` ("Format error decoding ..."), `ImageError::IoError`
(forwards `std::io::Error`), `ImageError::Unsupported` ("The image format ...
is not supported") all hit `decode` or `format` and route to
"could not decode image" — acceptable.

## Substring matching is structurally brittle

Three of the bail strings the implementer themselves wrote in this PR
series do not survive `sanitize_load_error`. Every miss is silent:
the error rewords slightly and the user category flips with no
compile-time signal.

A non-brittle alternative — even given anyhow's type-erased shape —
is to introduce a typed error category enum (or sentinel cause types
attached via `.context(LoadErrorKind::TooLarge)`) that the sanitizer
inspects via `err.downcast_ref()` rather than substring matching the
render. The bail call sites then change once and stay correct under
later rewordings. The decode-pipeline error sites are all in
warpui_core/src/image_cache.rs and asset_cache.rs — fully under our
control. There is no good reason to substring-match.

This is the single biggest correctness concern in the PR; the
substring map silently misroutes ~3 of ~10 production error strings
today.

## Other lens items

- **AssetState::FailedToLoad rewrite** logs raw via `log::warn!`,
  returns sanitized to user. Matches §182. ✓
- **`ImageType::Unrecognized` direct rewrite** (`"could not detect
  image format"`) is independent of the sanitizer; the strings are
  not piped through it. Noted, not a bug.
- **Index validity / lifecycle** — single-image v1 always has index 0.
  When `update_params` is called the View instance is the same; old
  spawn callbacks may fire after the images vector has been replaced
  but they go through `me.params.images.get_mut(index)`, which is
  bounded-checked. If the new vector is shorter the rewrite is
  silently dropped; if it is longer the old callback could write into
  an unrelated entry at the same index. In v1 single-image this is
  benign; for the multi-image follow-up (§ "next iterations") this
  becomes a real concern. Worth a comment near `update_params`
  flagging that re-entrancy under `update_params` can cross-write.
- **`update_image_at` ordering** (write slot, then start load) is
  fine; `start_asset_load` does not read `self.params.images[index]`
  synchronously, only the asset cache.
- **`start_asset_loads` collect→iterate** is correct; the order of
  asset loads now matches enumeration order, identical to the
  pre-refactor iter loop. No visible behavior change.
- **`asset_source.clone()` for the spawn closure** — a `String` alloc
  for `LocalFile`. Acceptable; one-shot per load.
- **TOCTOU on cache re-query** — re-querying `load_asset::<ImageType>`
  inside the callback is safe; the cache state is monotonic for a
  given key (Loading → Loaded | FailedToLoad), and `load_asset` on a
  reached terminal state is a lookup, not a re-fetch. ✓

## Test rigor — three of the three sanitize-category tests use
synthesized strings, not real bail strings

`sanitize_load_error_picks_too_large_category` uses `"local asset
exceeds size cap"` (a real string). ✓
`sanitize_load_error_picks_decode_category` uses `"png decode error:
invalid IHDR chunk"` — synthesized; never produced by the pipeline.
`sanitize_load_error_falls_back_to_read_category` uses `"io error:
connection reset"` — synthesized.

Had the tests pinned the actual strings emitted by 4a/4b/4c (e.g.
`"animated image has too many frames"`, `"svg dimensions exceed
render budget"`, `"Memory limit exceeded"`), they would have failed
loudly and surfaced both bugs above. Test-fixture strings should
mirror exactly what the production sites bail with.

Missing coverage:
- `Loaded { Png | Jpg | ... }` returns `None`.
- `Loading` returns `None`.
- `Initial` returns `None`.

These are cheap and worth adding.

# Suggestions

1. **Blocking — switch from substring matching to a typed-cause
   sanitizer.** Introduce an enum (e.g.
   `enum LoadErrorKind { TooLarge, Decode, Read }`) and attach it via
   `.context(LoadErrorKind::TooLarge)` (or a small sentinel error
   type) at every bail site in `image_cache.rs` and `asset_cache.rs`.
   `sanitize_load_error` then walks `err.chain()` and downcasts. This
   eliminates the entire class of re-categorization regressions.
2. **Blocking — fix the three concrete miscategorizations** even if
   the typed-cause refactor is deferred:
   - 4b: change `"too many frames"` to `"animated image exceeds frame
     budget"` (so it hits `exceeds`).
   - 4c: change `"svg dimensions exceed render budget"` to `"svg
     dimensions exceeds render budget"` (or to `"svg pixel area
     exceeds render budget"`).
   - For image-rs `Memory limit exceeded`: extend the substring set
     to `s.contains("exceeds") || s.contains("exceeded") ||
     s.contains("limit")`. (Note: matching `"limit"` would catch
     `LimitErrorKind::Unsupported` too, which is acceptable since
     unsupported-strict-limits is morally a too-large condition.)
3. **Should-have — pin sanitize-category tests against the real
   production strings**, not synthesized ones. Specifically add
   tests that consume each of: `"image is too large to preview"`,
   `"animated image has too many frames"`, `"animated image exceeds
   total pixel budget"`, `"svg dimensions exceed render budget"`,
   `"local asset exceeds size cap"`, `"local asset is not a regular
   file"`, plus `image-rs`'s `LimitError::DimensionError` and
   `LimitError::InsufficientMemory` Display strings. Each test
   asserts the expected category. This is the regression net the
   substring approach demands.
4. **Should-have — add the three negative-case tests** for
   `rewrite_image_for_load_state`: `Loaded { Png(...) }`,
   `AssetState::Loading`, `AssetState::Initial` all return `None`.
5. **Nit — add a comment near `LightboxView::update_params`** noting
   that pending spawn callbacks from a prior `params` may fire after
   the swap and that the bounds check is the only safeguard. v1 is
   single-image so this is benign, but the multi-image follow-up
   needs to either generation-tag the callback or invalidate the old
   entries.
6. **Nit — the operator log** uses `"GH9729: image preview load
   failed: {}"`. Consider `target: "image_preview"` for log-filter
   ergonomics; minor.
