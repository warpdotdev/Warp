# Findings

- The shape `if is_supported_image_file(path) { return FileTarget::ImagePreview; }` is the right idiom here. The existing function is already a sequence of guard-style early returns (steps 1-5: markdown, Warp editor, env editor, binary fall-through, then a final `match`), so a step-0 guard is consistent with its neighbours rather than gratuitously novel. Folding into the trailing `match` is not viable because that `match` runs after `is_openable_in_warp` is computed; image short-circuiting must happen before the binary classification, and a guard at the top expresses that constraint clearly.
- The 4-line comment is appropriate. It earns its keep because the reason the short-circuit must precede the existing chain is subtle: SVG is not in `crates/warp_util/src/file_type.rs`'s `is_binary_file` list (verified — `svg` is absent), so without the guard SVG flows through to `OpenableFileType::Text` and ultimately the code editor; PNG/JPEG/GIF/WebP are in `is_binary_file`, so they would otherwise hit step 4's `SystemGeneric` fall-through. The comment explains both branches in one breath. The §74 reference is helpful and properly scoped.
- The `notebooks/link.rs` bypass-removal is sensibly co-located with this commit. The commit-message argument is correct: with the resolver now classifying images as `ImagePreview`, the old `if is_supported_image_file(&path) { FileTarget::SystemGeneric } else { resolve_file_target(...) }` would shadow the resolver branch on the file-tree -> link path, making the resolver's image arm dead code on that route. Splitting the two changes would have created a transient state where the new resolver branch was unreachable from the link path. Tightly coupled; one commit is right.
- Test rigor for 1b on its own is appropriate. Verified `git show 256d31d` adds the three resolver tests prescribed in `tech.md` §613 (`resolve_file_target_image_preview_for_each_supported_extension`, `..._takes_precedence_over_markdown`, `..._non_image_binary_still_system_generic`) and updates the two existing binary tests that previously used `image.png` to `archive.zip`. So 1c does cover what 1b changed in `openable_file_type.rs`.
- The `notebooks/link.rs` bypass-removal had an existing test that exercised the old `SystemGeneric` branch — `test_open_local_image_uses_system_generic_target` in `app/src/notebooks/link_tests.rs:179`. After 1b's change this test would have continued to compile but assert the wrong target. It was updated in a follow-up commit `f743be1` ("update link test for new ImagePreview routing") that renames it to `test_open_local_image_uses_image_preview_target`, flips the assertion, and adds the missing `initialize_settings_for_tests` setup that the resolver path requires. That update is correct, but it should ideally have been part of commit `c15c1be`. This is a real but minor process nit — between `c15c1be` and `f743be1` the test was wrong; on a bisect that range would show a stale assertion. Counts as nit, not concern, because: (a) the spec authorizes 1c as the test commit, (b) the rename is a behavioural assertion swap not a structural fix, (c) git history shows the gap was closed inside the same branch before merge.
- The "currently text-classified, would otherwise open as raw XML in the code editor" claim for SVG is accurate against today's `is_file_openable_in_warp`. `crates/warp_util/src/file_type.rs:46-114` lists `jpg/jpeg/png/gif/bmp/tiff/tif/webp/ico` as binary; `svg` is conspicuously absent. So SVG returns `Some(OpenableFileType::Text)` from `is_file_openable_in_warp`, falls through markdown and code-editor checks, and lands at step 5 `External/SystemDefault` or step 2 `CodeEditor` depending on `EditorChoice` — i.e., raw XML in some editor. Comment is not brittle in a misleading way; it's brittle only in the trivial sense that if someone later adds `svg` to `is_binary_file`, the comment's first parenthetical will need amending. Worth living with.
- "step 0" numbering is fine. The function's existing comments are `// 1.` through `// 5.`, so `// 0.` slots in idiomatically as "happens before step 1" without renumbering. Renumbering to `1..=6` would have generated unnecessary diff churn and made the spec's §74 snippet (which uses `// 0.`) drift from the code.
- The diff in `notebooks/link.rs` also drops the `is_supported_image_file` import, which is correct and shows the cleanup was done thoroughly rather than leaving an unused import warning.
