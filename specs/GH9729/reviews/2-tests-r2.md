# Findings

- **Coverage maps 1:1 to spec bullets, with a justified deviation on the size-cap test.** The four tests at `app/src/workspace/view_test.rs:3024-3115` cover the exact bullet set from `tech.md` §613 lines 623-626: under-cap Resolved, over-cap Error (sanitized constant), nonexistent path Error, non-regular-file Error. The over-cap test deviates from the spec's literal "sparse `MAX_PREVIEW_FILE_BYTES + 1`" by passing a 100-byte cap to the helper and writing 200 bytes; the inline comment explicitly justifies this ("matches the spec's intent (file size > cap) without materializing the production 64 MB envelope"). This is exactly the right reason the helper was extracted with `max_bytes` as a parameter — promoting `MAX_PREVIEW_FILE_BYTES` to module scope and then ignoring it in the test would defeat the purpose. Defensible deviation.
- **Helper visibility correctness.** `build_image_preview_entry` is declared as `fn` (not `pub fn`) at module scope of `view.rs:23852`. `view_test.rs` is included as a child module via `#[path = "view_test.rs"] mod view_test;` from `view.rs:16`, so `super::build_image_preview_entry` from inside the child module reaches the parent module's free function without a `pub` visibility leak. Confirmed working: this is the cleanest way to expose a private helper to tests in this codebase. No notes.
- **`LightboxImageSource: Debug` confirmed.** `crates/ui_components/src/lightbox.rs:27` derives `Debug` on the enum, so the `panic!("expected Resolved with LocalFile, got {other:?}")` formatter is valid. `LightboxImage` (line 46) also derives `Debug`. The 3a/3b reviewers correctly flagged this elsewhere; here it is unambiguously satisfied.
- **Test naming drops the file's `test_` prefix convention.** Every one of the ~80+ existing tests in `view_test.rs` uses `fn test_*()` (verified via `grep -nE "^\s*fn test_"` — uniform across lines 305-3015). The four new tests use `image_preview_arm_*` with no `test_` prefix. This is the same drift R2 flagged on item 1c (`/Users/norfeldt/repos/warp/specs/GH9729/reviews/1c-r2.md` finding "Test naming drift"). Recommendation is the same: rename to `test_image_preview_arm_dispatches_*` for local-consistency. Mechanical rename, no behavior change. Not blocking, but the second occurrence in this branch elevates this from "nit" toward "pattern worth correcting before the branch closes."
- **`tempfile::TempDir::new().unwrap()` vs the file's `TempDir::new().expect("failed to create temp dir")` convention.** `view_test.rs:34` already imports `tempfile::TempDir` at module scope, and the existing six tempdir call sites (lines 313, 366, 434, 478, 2739, 2849) all use the bare `TempDir::new()` symbol with `.expect("failed to create temp dir")`. The new tests use the fully qualified `tempfile::TempDir::new()` and `.unwrap()`. Fully qualified path is redundant given the module-level `use`. `.unwrap()` vs `.expect()` is a style point — `.expect()` gives a slightly better failure message but `.unwrap()` is acceptable in tests. Minor consistency nit; recommend dropping the `tempfile::` prefix and either keeping `.unwrap()` or aligning to `.expect("failed to create temp dir")`. Not blocking.
- **`use std::io::Write as _;` placement inside test fns.** The two write-tests both have `use std::io::Write as _;` as the first line of the fn body. Hoisting to module scope would shave two lines and is consistent with the file's existing `use` block at lines 1-86. The `as _` rename is the right call to avoid leaking `Write` into other tests' scope, but at module scope a `#[cfg(feature = "local_fs")] use std::io::Write as _;` line would do the same job. Style preference; both work. Not flagging as blocking.
- **Inline `/* max_bytes */ 4096, /* max_message_len */ 256` parameter comments.** Idiomatic call-site disambiguation when the function takes multiple positional numeric args of the same primitive type. `build_image_preview_entry(path, 4096, 256)` is significantly less readable than the commented form. The codebase does not have a strong convention either way (rg shows ~12 occurrences of `/* … */` block comments at call sites across `app/src/workspace/`), so this is not aberrant. Worth keeping.
- **Refactor + tests in one commit is acceptable here, but borderline.** 32cee73 does three things: (1) promote `MAX_PREVIEW_FILE_BYTES` and `MAX_ERROR_MESSAGE_LEN` from arm-scope to module-scope `const`, (2) extract `build_image_preview_entry` from the arm body, (3) add four tests. The refactor exists to enable the tests, so they are causally coupled — splitting refactor and tests into separate commits would leave a transient state where the helper exists with no caller exercising it. The commit message explicitly enumerates the three steps in order and gives the spec rationale for each. The arm becomes a pure 5-line wrapper, which is a structural improvement independent of testability. Defensible single commit. R2 (2-arm) already noted this churn was foreseeable when 38aec6e declared the consts at arm scope; the right fix would have been declaring them at module scope from the start, not splitting 32cee73. Not blocking.
- **Dispatch wrapper is uncovered.** The arm body itself reduces to:
  ```rust
  let image = build_image_preview_entry(&path, MAX_PREVIEW_FILE_BYTES, MAX_ERROR_MESSAGE_LEN);
  ctx.dispatch_typed_action(&WorkspaceAction::OpenLightbox {
      images: vec![image],
      initial_index: 0,
  });
  ```
  None of the four tests exercise the `vec![image]`, `initial_index: 0`, or the `dispatch_typed_action` call path. Acceptable gap: this is a five-token wrapper around tested code, the `OpenLightbox` action shape is exercised elsewhere (artifacts/mod.rs caller, integration tests in item 5), and the test names ("`image_preview_arm_dispatches_…`") implicitly accept this scope by testing the entry construction rather than the dispatch. The 2-arm R2 review made the same call. Worth being explicit in the review log so a future reviewer doesn't read the test names literally and look for missing assertions.
- **`truncate_message` boundary branches remain dead in test coverage.** All four production error messages (`"image is too large to preview"` = 31 chars, `"could not read image"` = 20 chars, `"not a regular file"` = 18 chars) sit well under any realistic `max_message_len`. The four tests pass `max_message_len: 256`, well above the 31-char ceiling, so `truncate_message` always hits the `chars().count() <= max_len` early-return branch. The truncation path (lines 23898-23900) and the `max_len <= 1` degenerate path (lines 23895-23897) of `view.rs` are unreached by 32cee73. Same finding as 2-arm R2. The recommendation there ("two-line follow-up tests for boundary cases") still stands but should be tracked as a separate item, not against 32cee73 — the spec at §613 explicitly enumerated the four arm-tests and did not require `truncate_message` boundary tests.
- **No symlink test, no non-UTF-8 path test.** Spec §613:621-626 does not require either. `metadata` follows symlinks by design, so the test set covers symlink-to-regular-file and symlink-to-directory through the existing Resolved and non-regular-file tests transitively. `to_string_lossy()` on a non-UTF-8 path is exercised by the `Resolved` test (the path is UTF-8 here, but `lossy` is a no-op on UTF-8) — the lossy fallback branch is not directly tested. Not a 2-tests gap; if it matters, it's a 2-arm gap and 2-arm R2 already shipped.
- **Per-test inline comments are pulling weight.** The block comment at lines 3017-3022 explains spec ref and dispatch-vs-helper-scope rationale. The inline comment at line 3058 ("Write 200 bytes against a 100-byte cap") justifies the cap deviation. The inline comment at line 3091 ("Sanitized constant; the underlying io::ErrorKind::NotFound is logged via log::warn! and never interpolated here") encodes a security-review property. The block comment at lines 3098-3102 explains the FIFO-cross-platform decision and forwards FIFO coverage to §613:633. None of these are restating obvious behavior. Comment density is appropriate.
- **Test isolation is clean.** Each test creates its own `TempDir`, drops it at end-of-test (RAII cleanup). No shared state, no `#[serial]` requirement, parallelizable under `cargo nextest`. The metadata-fail test creates a `TempDir` it never writes into — that's deliberate (the parent dir exists, so the failure is purely "file does not exist" rather than "ENOENT on parent dir"), and the comment makes that intent explicit. Good rigor.
- **`#[cfg(feature = "local_fs")]` gate is correct.** All four tests touch the filesystem; gating on `local_fs` matches the convention used by other fs-touching tests in the file (e.g., `test_workspace_sessions_*` lines 962-1019). Without this gate the tests would fail to compile on the WASM/web build. No notes.

# Suggestions

- **(Naming consistency, mechanical)** Rename the four tests with the `test_` prefix to match the file's uniform convention: `test_image_preview_arm_dispatches_resolved_when_under_size_cap`, etc. This is the same recommendation R2 made on 1c; correcting it on both items closes the only convention drift introduced by GH9729.
- **(Style consistency, trivial)** Drop the `tempfile::` prefix on `TempDir::new()` (already imported at module scope) and align to either `.unwrap()` or the file's `.expect("failed to create temp dir")` pattern. Pick one and apply it to all four call sites.
- **(Style consistency, trivial)** Hoist `use std::io::Write as _;` to the module-scope `use` block at the top of `view_test.rs`, gated `#[cfg(feature = "local_fs")]`. Two of the four tests need it; the other two don't, but the `as _` rename means it does not pollute their scope. Removes two redundant lines.
- **(Coverage gap, follow-up — not against 32cee73)** Add boundary-case unit tests for `truncate_message` (`max_len = 0`, `max_len = 1`, exact-cap input, multi-byte char straddling the boundary). The §613 spec did not enumerate these, so they belong in a separate follow-up commit if at all. Currently the truncation branch is dead in test coverage; that's defensively fine since production callers pass short categorical constants, but the helper's behavior is asymmetric across `max_len <= 1` vs `> 1` and it would be cheap to lock it in.
- **(No change requested)** The single-commit refactor+test is acceptable. The dispatch-wrapper coverage gap is acceptable given the helper boundary the spec authorized at §613:621.
