# Findings

## Idiomatic Rust — solid

- The refactor of `start_asset_load` from associated function to `&mut self` method is clean and matches `tech.md` §182 verbatim ("Refactor `start_asset_load` to a method on `&mut self` that takes the entry index alongside the `AssetSource`"). The signature `(&mut self, index: usize, asset_source: AssetSource, ctx: &mut ViewContext<Self>)` is the right shape.
- `start_asset_loads` collecting `Vec<(usize, AssetSource)>` first to avoid aliasing `&self` with `&mut self` is the textbook borrow-checker workaround. `mem::take` / `take_mut` would not help here (we don't want to consume the Vec, just iterate it). The clone of `AssetSource` per entry is acceptable: `AssetSource` is `Clone, Hash, PartialEq, Eq, Debug` — designed to be cheaply cloneable.
- `update_image_at`'s reorder (read source out, write slot, then start load) reads cleanly. The early-return `if index >= self.params.images.len()` is appropriate.
- The `match &image.source { Resolved { asset_source } => Some(asset_source.clone()), _ => None }` form is idiomatic.
- `rewrite_image_for_load_state` is a clean pure helper returning `Option<LightboxImageSource>`. Free fn at module scope, no `self` — good for testing.
- `AssetState::FailedToLoad(Rc<anyhow::Error>)` confirmed at `crates/warpui_core/src/assets/asset_cache.rs:90-95`. Tests construct `Rc::new(anyhow::anyhow!(...))` correctly.
- `AssetState::Loaded { data: Rc<T> }` confirmed; tests use `Rc::new(ImageType::Unrecognized)` correctly.

## `sanitize_load_error` — substring matching is brittle and misclassifies real bails

This is the main concern. The helper does `format!("{err}").to_lowercase()` then `contains(...)`. anyhow's `Display` prints only the top-level message (not the full chain — that requires `{err:?}`), so currently with no `.context()` wrapping in the caller chain the substring lookup matches the bail strings directly. **However**, cross-checking against the actual bails introduced by 4a/4b/4c/5a:

| Actual bail string | Sanitize substring hit | Mapped category | Intended category |
|---|---|---|---|
| `image is too large to preview` (4a static, image_cache.rs:442) | "too large" | too-large ✓ | ✓ |
| `animated image exceeds total pixel budget` (4b, :392) | "exceeds" | too-large ✓ | ✓ |
| `animated image has too many frames` (4b, :385) | none | "could not read image" | should be too-large |
| `animated image has no frames` (4b, :398) | none | "could not read image" | could-not-decode is closer |
| `decode_animated_with_limits called with non-animated format` (:379) | "format" | "could not decode image" ✓ | ✓ (programmer-error path, but fine) |
| `svg dimensions exceed render budget` (4c, :481/485) | none — note "exceed" without trailing 's' | "could not read image" | should be too-large |
| `local asset is not a regular file` (5a, asset_cache.rs:537) | none | "could not read image" | ✓ (read-style failure is acceptable here) |
| `local asset exceeds size cap` (5a, :554) | "exceeds" | too-large ✓ | ✓ |

Two clear miscategorizations:
1. **"svg dimensions exceed render budget"** — the helper checks `"exceeds"` (with trailing s) but the bail says `"exceed"`. This silently miscategorizes the SVG dimension cap as "could not read image" instead of "image is too large to preview." Both Manual Validation §11 ("the per-entry error state ('svg dimensions exceed render budget')") and the categorical intent of §182 expect this to land in the too-large bucket.
2. **"animated image has too many frames"** — same misclassification problem; "too" alone doesn't match the `"too large"` substring, so it lands in "could not read image" rather than "image is too large to preview."

The tests in this commit only cover synthesized strings (`"local asset exceeds size cap"`, `"png decode error"`, `"io error: connection reset"`); they exercise the helper but **do not exercise the actual bail messages from 4a/4b/4c/5a**. R1 will likely flag this. A test using the real bail strings would have caught both miscategorizations.

The mitigation cost is low: extend the "too large" branch to also match `"exceed"` (which subsumes both `"exceeds"` and `"exceed render budget"`), and add `"too many"` to the same branch.

## anyhow Display chain semantics — currently OK, brittle to future refactors

anyhow's `Display` impl prints only the outermost message (the most recently added `.context()`); the full chain requires `format!("{err:?}")`. I verified there is no `.context()` / `.with_context()` wrapping anywhere in the asset_cache or image_cache bail paths today, so the top-level `Display` matches the bail string. If a future change adds context wrapping ("decoding asset" wrapping "image is too large to preview"), the substring match at the outer layer would silently slide everything into "could not read image" because the cap wording would be hidden in the source chain. Worth a brief comment in `sanitize_load_error` warning that future `.context()` wraps must preserve at least one of the substring tokens at the outermost layer, or this falls back to the read category.

## `log::warn!` location — acceptable

The `log::warn!` lives inside `rewrite_image_for_load_state`'s `FailedToLoad` arm, so it only fires when the rewrite fires. Since the rewrite is unconditional on `FailedToLoad`, every failure is logged exactly once with the raw error. Not duplicated. The `Unrecognized` arm does NOT log, which matches §613:660 ("for the file-tree Lightbox surface … rewriting the entry"); whether to log on Unrecognized too is a minor judgment call (it's a real load anomaly worth noting), but the current behavior is consistent with §182 which only mentions logging for FailedToLoad.

## Test rigor

5 tests, all passing. Two strong assertions in test 1 — that the message is sanitized to a known constant, that it does NOT contain the original path or OS error string. These directly verify the §182 promise. Tests 3-5 exercise each of the three categorical branches.

**Missing tests** (any one of these would have caught the substring miscategorizations above):
- `sanitize_load_error_picks_too_large_for_svg_dim_cap` — synthesize the actual `"svg dimensions exceed render budget"` and assert too-large.
- `sanitize_load_error_picks_too_large_for_too_many_frames` — synthesize `"animated image has too many frames"` and assert too-large.
- `rewrite_image_for_load_state_returns_none_for_loading` — assert `Loading { handle }` returns `None` (the spinner case the rewrite must NOT touch).
- `rewrite_image_for_load_state_returns_none_for_loaded_recognized` — assert `Loaded { data: Rc::new(ImageType::Png(...)) }` (or any non-Unrecognized variant) returns `None`. Construction may need a small fixture image; a `mem::transmute` is not warranted.
- `rewrite_image_for_load_state_returns_none_for_evicted` — assert `Evicted` returns `None`.

The "None branch" tests are particularly important because they are the most expensive bug class: a false positive rewrite (e.g. accidentally rewriting `Loading` to `Error`) would produce permanent error states for assets that are still loading. Currently nothing pins this contract.

## Test naming — consistent

Drops the `test_` prefix, matching the convention from item 1c onwards. `post_load_callback_*` and `sanitize_load_error_*` are descriptive snake_case. Good.

## Comment quality

Doc comments on `start_asset_load`, `sanitize_load_error`, and `rewrite_image_for_load_state` all reference §182. The bullet list in `rewrite_image_for_load_state`'s docstring is especially good — it documents both *why* each state triggers a rewrite (would otherwise spin forever) and the user-visible category. This is the level of context that survives codebase archaeology.

Note: the doc comments contain em-dashes (e.g., "mislabeled or unsupported bytes — would otherwise spin forever"). The user's global instructions ban em-dashes only in drafted prose intended for the user, not in code comments — these are fine to leave.

## Refactor justification — three concerns in one commit

The commit changes (a) `start_asset_load` signature, (b) callsite refactor in `start_asset_loads` and `update_image_at`, (c) callback wiring with re-query, (d) two new free helpers, (e) 5 tests. The signature change cascades: if `start_asset_load` becomes `&mut self`, the callsite Vec materialization and the callback closure rewrite are forced. Splitting "refactor → callback wiring → sanitizer + tests" would have produced an intermediate state where `start_asset_load` is `&mut self` but the callback still does nothing — a no-op refactor that is not independently buildable as a behavior change. The three concerns are genuinely coupled. **Acceptable as a single commit.** The commit message itself does a good job listing the four sub-changes.

# Suggestions

1. **Fix the substring map for SVG and frame-count caps.** Either widen `"exceeds"` to `"exceed"` (which catches both `"local asset exceeds size cap"` and `"svg dimensions exceed render budget"`) and add `"too many"` to the too-large branch, or adopt typed errors in a follow-up. The minimum fix:

   ```rust
   if s.contains("too large") || s.contains("exceed") || s.contains("too many") {
       "image is too large to preview"
   }
   ```

2. **Add tests using the real bail strings, not synthesized prose.** The five tests today never load `image_cache.rs:442/481/485` or `asset_cache.rs:537/554`. Strings copied verbatim into a small new test would have caught issue 1. At minimum:
   - `sanitize_picks_too_large_for_real_svg_dim_bail` with `"svg dimensions exceed render budget"`.
   - `sanitize_picks_too_large_for_real_animated_frame_bail` with `"animated image has too many frames"`.
   - `sanitize_picks_too_large_for_real_animated_pixel_bail` with `"animated image exceeds total pixel budget"` (sanity check on the existing happy path).

3. **Add None-branch tests for `rewrite_image_for_load_state`.** Even just one — that `Loaded { data: Rc::new(ImageType::Png(small_fixture)) }` or `Evicted` returns `None` — pins the most important contract: do not rewrite recognized loaded images.

4. **Add a brief warning comment on `sanitize_load_error`** noting the substring approach assumes the bail message reaches Display unwrapped. If a future caller adds `.context("decoding asset")?`, the substring match will fall through to the read category. Suggested wording:

   ```rust
   /// NOTE: This relies on the top-level `Display` of the error containing
   /// the bail substring. If callers begin wrapping with `.context(...)`,
   /// the substring map must move to `format!("{err:?}")` (the full chain)
   /// or be replaced by typed errors.
   ```

5. **Optional**: log `Unrecognized` rewrites at `log::info!` so operators can see the mislabeled-file case fire in the field. Currently silent.

6. **Optional**: extract the three categorical strings to module-level constants (`MSG_TOO_LARGE`, `MSG_DECODE`, `MSG_READ`). The lightbox render path's downstream test coverage is then resilient to message-text edits.
