# Findings

- [nit] The no-op arm in `app/src/workspace/view.rs` was correct and unreachable as of fa5336b, but both 1b (commit `c15c1be`, resolver short-circuit) and 2b (commit `38aec6e`, OpenLightbox dispatch) have since landed on this branch. The stub arm was correctly replaced by `38aec6e` rather than left as dead code, so this is purely a "the comment in the original commit message accurately predicted the future" observation, not a live defect. No action needed at this point in the loop.
- [minor] Two non-pattern-match sites in the codebase construct `FileTarget::SystemGeneric` specifically for image files, bypassing the new variant entirely:
  - `app/src/ai/blocklist/block.rs:274` — `is_supported_image_file(absolute_path).then_some(FileTarget::SystemGeneric)`
  - `app/src/ai/ai_document_view.rs:919-920` — `if is_supported_image_file(path) { FileTarget::SystemGeneric } else { resolve_file_target(...) }`
  These are constructors, not matches, so they cannot be caught by the §74 "grep for `matches!`" audit, and they are technically out of scope for 1a (which only adds the variant). However, they will continue to route image clicks from AI blocks / AI document view to the OS instead of the new Lightbox even after 1b/2b land. They should be tracked as a follow-up audit (or filed under a later TODO item) so the v1 routing contract is consistent across all entrypoints, not just the file-tree path. Flagging here because §74's audit note says the goal is that "`ImagePreview` is treated as in-Warp, do not hand off to OS" — these two sites violate the inverse of that goal by hard-coding OS handoff for images.
- [nit] The doc comment on the new variant in `openable_file_type.rs` is accurate but could note that the resolver short-circuit (1b) is the production source of this variant, mirroring the comment style on `MarkdownViewer`/`CodeEditor`. Pure cosmetic.
