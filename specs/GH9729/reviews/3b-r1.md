# Findings

## Spec fidelity

- The rendered shape matches §182 line 195 line-by-line. The arm produces a
  centered Flex column (`CrossAxisAlignment::Center`), filename
  (`current_description`) on the first line when present, then sanitized
  `message` on the second line, with `DESCRIPTION_SPACING` between them. No new
  style tokens introduced; `appearance.ui_font_family()`, `text_size`
  (`lightbox_text_size`), and `ColorU::white()` are all reused.
- §182 line 195 also says "Style consistent with the existing close-button
  panel; use the existing UI tokens." The arm reuses the same
  `ui_font_family`/`text_size`/white pairing used by the description text in
  `content_with_description` (lightbox.rs:211-213). Consistent.

## Scope narrowing — justified

- §182 line 209 contains the literal text:
  > "Adopting `Error` at the artifacts site is listed as a small follow-up."
- This explicitly authorizes deferring `app/src/ai/artifacts/mod.rs:362-365`.
  The original TODO bullet at IMPLEMENTATION_TODO.md was therefore narrower
  than §182 prescribed; the rewrite to "Render only — adopting Error at
  artifacts/mod.rs is a follow-up per §182" actually realigns the bullet with
  the spec, rather than descoping below it. Not a cover-descope.
- However, no follow-up bullet exists yet in IMPLEMENTATION_TODO.md for the
  artifacts adoption; without it, the §182 follow-up is invisible to the
  tracker. (Nit, see Suggestions.)

## Match-arm placement

- Order in `Lightbox::render` (lightbox.rs:157-205) is now:
  1. `(Some(Resolved), Some(native_size))` — render image
  2. `(Some(Error { message }), _)` — render error panel  *(new)*
  3. `_ if image_count == 0` — "No images"
  4. `_` — `loading_element`
- The new `Error` arm is correctly placed BEFORE the catch-all `_`, as the
  commit message claims. If it had been placed after, the catch-all `_` would
  have shadowed it.
- `(Some(Loading), _)` correctly falls through to the catch-all `loading_element`.
- `(Some(Resolved), None)` (image bytes still being decoded) also correctly
  falls through to `loading_element`. The introduction of the Error arm does
  not alter Loading-state behavior.

## Non-blocking dismissal — correct (intentional asymmetry with Resolved)

- The new arm calls `column.finish()` without wrapping it in `EventHandler`
  with `on_left_mouse_down(StopPropagation)`. This is the deliberate inverse
  of the `Resolved` arm at lightbox.rs:170-172, which DOES stop click
  propagation precisely so that clicks on the image don't dismiss the lightbox.
- For `Error`, §182 line 195 is explicit: *"The panel does not block
  dismissal; Escape / scrim / × continue to work."* The omission of
  `StopPropagation` is the correct realization of that requirement — clicking
  on the error text deliberately bubbles to the scrim's `Dismiss` handler at
  lightbox.rs:228-231. The user can dismiss by clicking anywhere, including
  *through* the error panel.
- No focus/keyboard capture is introduced. The arm is pure layout; Escape
  binding is delivered via the existing `dismiss_keystroke` plumbing on the
  close button at lightbox.rs:148. No regression there.

## Sanitization — defense layering OK, message bound is upstream-only

- The render code uses `message.clone()` directly with no re-sanitization or
  truncation. The §179 contract makes callers pre-sanitize; verified at
  every existing construction site:
  - `app/src/workspace/view.rs:23875-23876` truncates via
    `truncate_message(message, MAX_ERROR_MESSAGE_LEN /* 256 */)` before
    constructing the variant.
  - `app/src/workspace/lightbox_view.rs:198-205` constructs from either a
    static literal `"could not detect image format"` or the `&'static str`
    return of `sanitize_load_error(err)` — both bounded categorical constants.
- So today no Error site can produce >256 chars, and the render code's
  no-defensive-cap stance is sound for the v1 surface.
- BUT the §182 follow-up (artifacts/mod.rs adoption) is outside
  `MAX_ERROR_MESSAGE_LEN`'s reach: that constant lives in
  `app/src/workspace/view.rs:23837` and is not exported. A future implementer
  adopting `Error` at the artifacts site could easily forget to truncate. The
  invariant "every Error construction is pre-truncated" is enforced by
  convention, not by type. (See Suggestions.)
- Render-side defensive truncation was not added; that is consistent with
  §179's "callers must sanitize" model and avoids double-truncation, but it
  does mean the invariant is purely upstream.

## Description leakage — clean

- §175 ("filename, not absolute path") is upheld. At construction:
  - `view.rs:23853`: `path.file_name().map(|n| n.to_string_lossy().into_owned())`
    — last path component only, never the full path.
  - `lightbox_view.rs:198/204`: rewrites only the `source` field; the
    `description` is whatever was set at the original entry construction
    (which already came from `path.file_name()` upstream).
- No call site passes `path.to_string_lossy()` into `description`. Confirmed.

## Long-message rendering — safe at v1, fragile for the follow-up

- Rendering a 5000-char `message` would produce a large Text widget that
  could occlude the × close button. Today this cannot happen:
  - Categorical constants (`"could not read image"`, `"could not decode
    image"`, `"image is too large to preview"`, `"could not detect image
    format"`, `"not a regular file"`) are all <40 chars.
  - `truncate_message` caps the workspace arm at 256 chars.
- Future-fragile: a renderer-side defensive cap (or moving
  `MAX_ERROR_MESSAGE_LEN` into `lightbox.rs` and applying it at render time)
  would convert the invariant from convention to type-enforced. Not blocking;
  noting for the §182 follow-up.

## Accessibility

- §211-215 explicitly descopes the GPUI a11y plumbing from v1. The Error
  panel inherits the same a11y posture as the rest of the Lightbox: the
  filename is rendered as visible text (announced when focus enters that
  text node) but is not exposed as the image's accessible label. No
  regression introduced by this commit; nothing §182 required here.
- Keyboard: Escape continues to dismiss via the close button's
  `dismiss_keystroke` binding. No focus trap is introduced (the Error column
  is non-focusable layout). Good.

# Suggestions

1. (Nit, tracker hygiene) Add a separate IMPLEMENTATION_TODO.md follow-up
   bullet for "Adopt `LightboxImageSource::Error` at
   `app/src/ai/artifacts/mod.rs:362-365`" with the §182 reference, so the
   follow-up survives the bullet rewrite. Currently the rewrite removes the
   only mention of artifacts/mod.rs from the tracker.

2. (Nit, future-fragile invariant) When 3b's follow-up adopts `Error` at the
   artifacts site, either (a) export `MAX_ERROR_MESSAGE_LEN` /
   `truncate_message` from `app/src/workspace/view.rs` for reuse, (b) move
   them into `crates/ui_components/src/lightbox.rs` and apply at render
   time, or (c) at minimum, document the "callers must truncate" invariant
   on the `LightboxImageSource::Error` variant's doc comment. The current
   variant's docs (in commit 3a) should already carry this; if not, that's
   a minor doc nit for 3a, not 3b.

3. (Optional, non-blocking) If a future change wanted Error-panel clicks NOT
   to dismiss (e.g. so users can select-and-copy the message), a single
   `EventHandler::new(...).on_left_mouse_down(StopPropagation)` wrap on the
   message-only Text would do it without affecting Escape / × / scrim
   outside the panel. §182 line 195 forbids this for v1, so leave as-is —
   noting only because the asymmetry with the Resolved arm may invite
   review questions later.
