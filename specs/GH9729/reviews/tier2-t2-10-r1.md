# Spec

§182 (tech.md, change 3 — surfacing the `Error` variant) reads:

> In `app/src/workspace/lightbox_view.rs`, refactor the asset-load callback shape so it can rewrite the entry on failure or on `Unrecognized`. ... Inside the callback, after `ctx.notify()`, look up the asset state via `AssetCache::as_ref(ctx).load_asset::<ImageType>(asset_source)`. The state transitions are observable as: `AssetState::Loaded { data: ImageType::Unrecognized }` → rewrite to `LightboxImageSource::Error { message: "could not detect image format" }`. This closes the rough edge where a mislabeled file (e.g. a `.png` containing tarball bytes) currently caches `Unrecognized` and renders a permanent spinner. `AssetState::FailedToLoad(err)` → rewrite to `LightboxImageSource::Error { message: sanitize(err) }` ... All other states leave the entry unchanged.

§695 (Follow-ups) reads:

> Convert `ImageType::Unrecognized` to `Err` globally with an audit of every `try_from_bytes` caller. Closes the mislabeled-file rough edge for surfaces other than the Lightbox file-tree path (which v1 already closes locally).

t2-4 shipped §695 (`Unrecognized → Err`). The combined contract is that mislabeled bytes now surface synchronously as `AssetState::FailedToLoad` for cache hits and inline-resolved `Bundled`/`Raw` sources, and the Lightbox must still render the §182 `Error` variant in that case. The t2-4 spawn-callback rewrite was the only enforcement point, and it never fires for `FailedToLoad` resolved before the `Loading` arm sets up the future — t2-10 closes that gap.

# Findings

- **[major→resolved] Bug is real and the fix is correct.** Pre-fix `start_asset_load` (commit `7780d31`, lines 145-168) gates the entire rewrite under `if let AssetState::Loading { handle } = ...`. Any state the cache resolves synchronously — `Loaded`, `Evicted`, `FailedToLoad`, or `Loading` with a `None` `when_loaded` — falls through with no rewrite installed. `AssetCache::create_future_for_loading_asset` (asset_cache.rs:391-410) returns `future::ready(())` for non-`Loading` internal states and `None` for an unknown handle, so a synchronously-`FailedToLoad` entry has no path to ever re-fire the rewrite. Post-fix the inline `apply_rewrite_to_slot` runs against the initial `load_asset` return value and `return`s before the `Loading` arm — for `FailedToLoad` the slot mutates to `Error` and the spawn is skipped; for `Loading` the helper returns `false` (no-op) and the existing async path is intact. Correct.
- **[minor] Synchronous-FailedToLoad reachability for the v1 file-tree repro path is *not* via the first `load_asset` call.** `Workspace::open_lightbox` builds `LightboxImageSource::Resolved { asset_source: AssetSource::LocalFile { path } }` (view.rs:23895). `AssetCache::load_asset` for `AssetSource::LocalFile` (asset_cache.rs:320-326) *always* takes the `load_asynchronously` path on a cold cache — it inserts `AssetStateInternal::loading()` synchronously and then spawns the read on the background executor. So `06-mislabeled.png`'s first open does *not* return `FailedToLoad` inline; it goes through the async path and the t2-4 spawn callback should have rewritten it. The commit message ("the asset cache can deliver `FailedToLoad` inline on the first `load_asset` call ... before any executor turn") is therefore not quite accurate for the cold-cache `LocalFile` path. The synchronous-`FailedToLoad` *is* reachable via (a) a warm cache hit on a re-open of the same path, (b) `Bundled` and `Raw` sources, which resolve inline (asset_cache.rs:300-319, 327-335), and (c) `update_params` calls on a `LightboxView` where the entry was loaded by an earlier session. The fix is still correct and the §182/§695 contract still requires this branch — but the headline repro story understates the analysis. I would expect the next-pass commit message to either point at the warm-cache path or include the actual stack from the manual repro session.
- **[minor] Test 2 (`leaves_loading_state_alone`) tests `Evicted`, not `Loading`.** The function-level comment is candid: "`AssetState::Evicted` is the cheapest non-Loading, non-FailedToLoad variant to construct in a unit test." This is fine as a coverage-of-the-no-rewrite-branch test, but it does not pin the §182 contract that the `Loading` arm specifically must not be rewritten. Since `rewrite_image_for_load_state` matches on `FailedToLoad` only with a catch-all `_ => None`, the contract is structurally enforced today — but a future tweak that adds an explicit `Loading` arm would not be caught by this test (it pins `Evicted`, not `Loading`). The test name slightly oversells the coverage. A second assertion in the same test, building an `AssetState::Loading { handle: AssetHandle { source, asset_type: TypeId::of::<ImageType>() } }` directly (the constructor is `pub` per the struct visibility — verify locally), would tighten this. Minor.
- **[nit] No "synchronous Loaded does not rewrite" test.** The §695 contract is that *only* `FailedToLoad` rewrites and `Loaded` must pass through (so the Lightbox renderer reads the cache directly). The existing `apply_rewrite_to_slot_leaves_loading_state_alone` covers `Evicted` and the `_ => None` catch-all happens to cover `Loaded` too, but a dedicated test would pin the §695 contract more explicitly. `AssetState::Loaded { data: Rc<ImageType> }` requires materialising an `ImageType` value, which is cheap for a 1×1 SVG or for `Rc::new(ImageType::StaticBitmap(...))` if a constructor exists — worth a follow-up test if the test-fixture cost is not prohibitive.
- **[nit] §182's "all other states leave the entry unchanged" includes a `Loading` slot that was already rewritten by a prior call.** Not a real concern — `apply_rewrite_to_slot` only mutates if `rewrite_image_for_load_state` returns `Some`, and that's only for `FailedToLoad`. No double-rewrite path: the inline branch mutates and returns; the spawn branch only runs for `Loading` (i.e., the inline call was a no-op); the spawn fires at most once per `when_loaded`. Confirmed clean.
- **[nit] Inline-rewrite docstring says "synchronously-`FailedToLoad`, which is the case the `Loading`-only spawn below cannot rescue."** Accurate, but a one-line "warm-cache hit OR `Bundled`/`Raw` source — see asset_cache.rs:300-340" pointer would make the next reviewer's life easier. The current docstring leans on the (slightly-inaccurate) commit-message narrative.

# What I checked

- `git show af7d5f5` and `git show 7780d31:app/src/workspace/lightbox_view.rs` to diff `start_asset_load` pre- and post-fix; confirmed pre-fix gates the entire rewrite under the `Loading` arm and post-fix calls `apply_rewrite_to_slot` inline + returns early on mutate.
- `AssetCache::load_asset` (asset_cache.rs:284-340) per source type: `Async`, `Bundled`, `LocalFile`, `Raw`. Confirmed `LocalFile` always inserts `loading()` and spawns via `load_asynchronously` on a cold cache — the synchronous-`FailedToLoad` reach is via warm-cache hits, `Bundled`, or `Raw`.
- `AssetHandle::when_loaded` (asset_cache.rs:108-110) → `create_future_for_loading_asset` (asset_cache.rs:385-411): for `Loading` returns a receiver-backed future; for `Loaded`/`Evicted`/`Error` returns `future::ready(())`; for an unknown handle returns `None`. Confirmed an already-failed entry will not retrigger the spawn callback.
- `rewrite_image_for_load_state` arm coverage (lightbox_view.rs:191-215 pre-fix): only `FailedToLoad` returns `Some`; all other arms (`Loaded`/`Loading`/`Evicted`) fall through `_ => None`. The inline `apply_rewrite_to_slot` therefore is correctly a no-op for `Loaded`/`Loading`/`Evicted` and rewrites `FailedToLoad` — matching §182.
- Double-rewrite analysis: inline branch `return`s after mutating; spawn branch is only entered for `Loading`; the spawn future fires exactly once. No double-rewrite path.
- `apply_rewrite_to_slot` index validity: both the early-return `Some(slot)` guard and the `Some(new_source)` guard tolerate out-of-bounds; the spawn callback now calls the same helper rather than `images.get_mut(index)` directly. Matches the test `apply_rewrite_to_slot_tolerates_out_of_bounds_index`.
- Test mod imports (lightbox_view.rs:443-445): `use super::*;` brings `AssetState`, `ImageType`, and the helper into scope; `Rc` is imported separately. The three new tests compile cleanly given the surrounding fixtures.
- `Workspace::open_lightbox` (view.rs:7468) → `LightboxView::new` (lightbox_view.rs:110) → `start_asset_loads` → `start_asset_load`. Asset source for the v1 file-tree path is `LocalFile { path }` (view.rs:23895), which lands on the async branch on a cold cache (see findings nit above).
- §182 and §695 read against the current change: behavioural contract is upheld; t2-10 is a strict superset of the t2-4 rewrite coverage.
