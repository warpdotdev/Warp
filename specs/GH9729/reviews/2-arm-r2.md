# Findings

- **Implementation matches the spec verbatim.** The arm body in `app/src/workspace/view.rs:5817-5872` is essentially the code block from `tech.md` §119 (lines 130-180) verbatim, including the four-armed `match std::fs::metadata(&path)` with two guard arms, the `Result<(), &'static str>` shape, the `LightboxImage` construction, and the single-element `vec![image]` dispatch. The spec endorses inline shape; extraction into a helper is not required at 2-arm. (32cee73 then extracts `build_image_preview_entry` for testability per §613, which is the right time to do it.)
- **`match` guard ordering is correct.** First-match-wins semantics handle `!is_file()` first, then the size cap, then the success arm, then the IO error. This ordering is important: directories and FIFOs can have arbitrary `len()` values that would falsely trip the size cap if the order were reversed. The shape is more readable than a chained `if let Ok(meta) = ... { if !meta.is_file() { ... } else if meta.len() > ... }` because the `Err` branch lives at the same indentation as the success guards.
- **`dispatch_typed_action` idiom matches `artifacts/mod.rs:311`.** Same `&WorkspaceAction::OpenLightbox { images, initial_index }` shape, same `vec!` construction style. Consistent with the only other production caller.
- **Module-scope const placement (at 38aec6e) is suboptimal but spec-aligned.** `MAX_PREVIEW_FILE_BYTES` and `MAX_ERROR_MESSAGE_LEN` are declared as block-scope `const` items inside the arm. The spec's example block at §119:121-122 also declares them inline at arm scope, so 38aec6e faithfully implements the spec. They become unreachable from tests, which 32cee73 then has to fix by promoting them to module scope. Given that 2-tests was already enumerated in the TODO at the time of 38aec6e, declaring them at module scope from the start would have avoided the churn — minor process nit, not a code-quality defect.
- **`truncate_message` placement at 38aec6e is far from its caller.** Caller at line 5863, definition at line 23882 — an ~18,000-line gap in the same file. Module-scope is fine, but readers diffing this commit have to jump end-to-end. 32cee73 implicitly fixes this by moving the helper next to the new `build_image_preview_entry`, so the issue is self-correcting one commit later. Not blocking, but a small code-locality hit on 38aec6e in isolation.
- **`truncate_message` correctness is fine; shape is slightly redundant.** `message.chars().count() <= max_len` is O(n) and is the only branch that traverses the full string — for the 256-char ceiling on short categorical constants this is irrelevant. A shorter form `message.chars().take(max_len + 1).count() > max_len` would short-circuit, but adds cognitive load for no measurable benefit at the actual call sites. The `max_len <= 1` branch is correct: at `max_len == 0` it produces `""` (zero chars taken); at `max_len == 1` it produces the first char (no ellipsis, since adding `…` would exceed the cap). The ellipsis at U+2026 is a single scalar value so the boundary is clean. Verdict: correct and defensible as written.
- **`truncate_message` is NOT directly unit-tested in 32cee73.** The four tests in `view_test.rs:3017-3115` exercise `build_image_preview_entry` end-to-end and rely on the production constants `"image is too large to preview"`, `"could not read image"`, `"not a regular file"` — all 31 chars or shorter, so they only exercise the `chars().count() <= max_len` early-return branch. The truncation branch and the `max_len <= 1` degenerate branch have **zero direct coverage**. This is acceptable defensively (the helper is dead code on the happy path) but if 4-svg or downstream code starts passing longer messages, the truncation behavior is currently unverified. Recommend a small follow-up unit test for boundary inputs (`max_len = 0`, `max_len = 1`, an exactly-`max_len`-char input, a multi-byte char straddling the cap).
- **Test rigor of 2-tests, otherwise.** 32cee73's four tests cover the dispatch shape correctly: each test asserts the exact `LightboxImageSource` variant and the `description` (filename) propagation. Three of the four metadata branches are covered (`Ok` regular file, `Ok` non-regular = directory, `Err` = nonexistent). The size-cap branch (`Ok(meta) if meta.len() > MAX`) is covered via the small-cap override (200 bytes vs 100-byte cap), which is exactly the right pattern for a 64 MB envelope. **Not covered**: the single-element `Vec` and `initial_index: 0` dispatch shape — that is the responsibility of telemetry/integration tests, and `build_image_preview_entry` returns a single `LightboxImage` rather than the full `Vec`, so the dispatch shape is a one-liner in the arm itself. Acceptable boundary; the integration coverage from item 5 backs this up.
- **`use` statement placement is non-idiomatic.** The two new `use` lines are inserted between `use super::WorkspaceRegistry;` and `use crate::ai::execution_profiles::editor::...`, breaking up the existing `super::` -> `crate::` grouping. House style in the file groups external crates (`use ui_components::...`, `use warpui::...`) together, separated from `use crate::...` and `use super::...` blocks. Pre-existing style in this file is already inconsistent (e.g., `serde_json` and `warpui::notification` appear before `use super::...` blocks at lines 143-144), so this is not a regression so much as a missed opportunity to restore order. Recommend grouping `ui_components::lightbox` and `warpui::assets::asset_cache` near other `warpui::*` and `ui_components::*` imports rather than wedged after `super::WorkspaceRegistry`.
- **Comment density is high but justified.** ~15 lines of comments for ~40 lines of code is heavy by normal standards, but: (a) this is a security-critical pre-read gate where each rejection branch encodes a threat-model decision, (b) the comments document *why* (sanitize-OS-error, cap-before-read, follows-symlinks-rejects-character-devices) rather than *what*, (c) §119 explicitly calls out these rationales as load-bearing. The comments are pulling weight, not restating obvious behavior.
- **String literals as inline constants.** `"not a regular file"`, `"image is too large to preview"`, `"could not read image"` are inline `&'static str`s. They could be promoted to named constants but the entire set is three strings used at exactly one call site each — promoting them adds indirection without measurable benefit. Acceptable.
- **`filename` carrier shape.** `path.file_name().map(|n| n.to_string_lossy().into_owned())` — correct use of `to_string_lossy` for a UI-bound string, correct `into_owned()` to detach from the `Path` borrow, correct `Option<String>` to allow root paths to render with no description. No notes.
- **`path.to_string_lossy().into_owned()` for `AssetSource::LocalFile { path: String }`.** Matches `AssetSource`'s API contract; lossy is the right choice for non-UTF-8 paths since `LocalFile` already takes `String`. No notes.
