# Findings

- **Spec fidelity (constants).** All four spec values land verbatim:
  `MAX_DECODE_DIMENSION = 8_192`, `MAX_DECODE_PIXELS = 67_108_864`,
  `MAX_DECODE_ALLOC = 256 * 1024 * 1024`. `decode_limits()` builds
  `image::Limits` exactly as the spec block at ¬ß234 specifies (default
  + per-axis + max_alloc). No deviation.

- **Spec fidelity (helper shape).** `decode_static_with_limits` is a
  faithful translation of the ¬ß234 pseudocode ‚Äî same `ImageReader::with_format`
  + `reader.limits(...)` + post-decode `pixels > MAX_DECODE_PIXELS` bail.
  The bail message string `"image is too large to preview"` matches the
  user-facing string the spec calls out (and matches the workspace `Error`
  variant message in `lightbox_view.rs::sanitize_load_error`, so item 5b
  surfaces it correctly ‚Äî see "What I checked" below).

- **`Limits` struct is `#[non_exhaustive]`.** Verified at
  `image-0.25.9/src/io/limits.rs:32`. The code constructs it via
  `Limits::default()` then assigns three fields, which is the
  forward-compatible pattern (a future image-rs adding fields cannot
  break this code, and a future stricter default is preserved by us
  only over-relaxing the three fields we name). No risk.

- **Per-axis check vs `max_alloc` enforcement on JPEG / PNG / WebP-static.**
  Walked the decode chain in `image-0.25.9`:
  - `ImageReader::decode()` (`io/image_reader_type.rs:311`) calls
    `limits.reserve(decoder.total_bytes())` BEFORE the buffer is
    allocated, and then `decoder.set_limits(limits)`. So `max_alloc` is
    in fact checked against the predicted output buffer for every
    decoder that reports `total_bytes()` correctly ‚Äî this is the
    primary defense for the "small file decompresses to huge buffer"
    bomb class.
  - PNG (`codecs/png.rs:265`) and JPEG (`codecs/jpeg/decoder.rs:168`)
    both override `set_limits` and do `check_dimensions` against the
    declared header dims. PNG explicitly notes it cannot constrain
    the inner `png` crate's intermediate allocations, so the
    `reserve(total_bytes())` step in `ImageReader::decode()` is what
    actually matters there.
  - WebP (`codecs/webp/decoder.rs`) does NOT override `set_limits`,
    so it uses the trait default at `io/decoder.rs:115` ‚Äî which still
    does `check_dimensions`. Combined with the upstream
    `reserve(total_bytes())` step in `ImageReader::decode()`,
    static WebP IS bounded on both axis and alloc through the
    helper. This is non-obvious but correct, and the spec at ¬ß234
    is silent on it; not a defect.

- **Bomb-class coverage:**
  1. 1√ó1_000_000_000 ‚Äî fails on per-axis height check before any
     buffer allocation. ‚úì
  2. Streaming progressive JPEG ‚Äî JPEG decoder honors per-axis
     limits via `check_dimensions` in its `set_limits` override.
     Combined with `reserve(total_bytes())`, alloc-bomb is also
     refused before bulk decode. ‚úì
  3. Compressed bomb (PNG zlib bomb) ‚Äî `reserve(total_bytes())` in
     `ImageReader::decode()` rejects oversized output buffers
     based on dimensions √ó bytes-per-pixel. Caveat: this checks
     OUTPUT size, not inner-decoder scratch buffers, but the latter
     are bounded by header dimensions (already capped at 8192).
     The PNG decoder's TODO comment about not yet calling
     `png::Reader::change_limits()` is a known image-rs gap; the
     post-decode `pixels > MAX_DECODE_PIXELS` belt-and-suspenders
     check is the second line of defense. ‚úì
  4. Per-axis cheat (8192√ó8192) ‚Äî exactly at cap. `check_dimensions`
     uses `>`, post-decode check uses `>`. So 8192√ó8192 = 67M
     pixels passes both, matching the spec's "MAX_DECODE_PIXELS = 67M"
     intent (the cap is inclusive). ‚úì
  5. APNG ‚Äî `PngDecoder::is_apng()` exists but `try_from_bytes`
     never calls it; the PNG branch goes through `ImageReader::decode()`,
     which produces a single `DynamicImage` for the default IDAT.
     APNGs therefore render as a still (existing pre-commit behavior
     unchanged). The dimension cap still applies because the PNG
     header dims gate everything. Not a regression. Not a security
     issue. Worth a comment in the spec follow-ups, not blocking here.
  6. Animated WebP probe ‚Äî `WebPDecoder::has_animation()` reads
     header bits; if it lies and a malformed file flips the wrong
     way, the static branch decodes it under the helper (capped).
     If `WebPDecoder::new` itself fails, `?` propagates. Safe.
  7. GIF ‚Äî confirmed unchanged in this diff (`Ok(ImageFormat::Gif)`
     branch is untouched). Item 4b territory.

- **`drop(decoder)` on the WebP-static branch.** Verified: this is
  cosmetic, not load-bearing. `WebPDecoder::new(Cursor::new(data))`
  takes the cursor by value; the inner cursor borrows `data: &[u8]`
  immutably. A second `Cursor::new(data)` inside the helper is an
  independent immutable borrow and would compile fine without the
  drop. The explicit `drop(decoder)` is defensible as a documentation
  device (the comment immediately above tells the reader why we
  re-open) but the code would be one line shorter without it.
  **Nit.**

- **`saturating_mul` on (u32 √ó u32) into u64.** As called out in the
  prompt, `(u32 as u64) * (u32 as u64)` cannot overflow `u64`. The
  saturating call is harmless belt-and-suspenders and arguably reads
  as more careful than `*`. Fine.

- **Catch-all branch.** The `_ => Ok(ImageType::Unrecognized)` arm
  short-circuits unsupported formats (TIFF, BMP, DDS, HDR, ‚Ä¶) before
  they reach any decoder ‚Äî decode-bomb resistance for those is
  vacuously satisfied because we never decode. The lightbox callback
  (item 5b) rewrites `ImageType::Unrecognized` to a user-facing
  `"could not detect image format"` string, so this is
  defense-in-depth rather than a hole.

- **Match arm coverage.** Read the surrounding `try_from_bytes`
  (image_cache.rs:319-end at commit). The static-raster arms updated
  in this commit are JPEG, PNG, and WebP-static ‚Äî exactly the three
  the spec ¬ß234 names. SVG runs above the match (`if data.first() ==
  Some(&b'<')`), `warp-img:` is hand-rolled (no `image::ImageReader`
  involvement, so `Limits` doesn't apply; size is constrained by an
  explicit `expected_byte_count` check on the header ‚Äî pre-existing).
  GIF and animated-WebP arms are correctly out of scope.

- **End-to-end error surface.** Confirmed
  `app/src/workspace/lightbox_view.rs::sanitize_load_error`
  categorizes by lowercase substring match: `"too large"` ‚Üí
  `"image is too large to preview"`. The bail string this commit
  emits contains `"too large"`, so the load callback rewrites
  `FailedToLoad` to the correct user-facing string. The path is
  closed.

# Suggestions

- **Drop the `drop(decoder);`.** It's cosmetic. If the team prefers
  keeping the documenting comment, replace with:

      // The has_animation() probe is the only reason we constructed
      // a WebPDecoder. Re-open through the limits-aware helper
      // (one extra header parse, negligible).
      let img = decode_static_with_limits(data, image::ImageFormat::WebP)?;

  No `drop` line. Either is acceptable; the current form is fine.

- **Inline-comment the WebP alloc-cap path.** Worth a one-line
  comment in `decode_static_with_limits` noting that for WebP the
  alloc cap is enforced by `ImageReader::decode()`'s
  `reserve(total_bytes())` step (since `WebPDecoder` does not override
  `set_limits`). Not strictly necessary, but a future reader who
  greps for "WebP" + "max_alloc" will otherwise wonder. Leave as-is
  if the spec covers it elsewhere.

- **APNG follow-up (informational, not blocking 4a).** APNGs hit the
  static-PNG branch and render as a still. Pre-existing behavior;
  the dimension cap still bounds the per-frame envelope so this is
  not a decode-bomb. If the team wants APNG to animate, that's a
  separate work item; if they want APNG explicitly rejected, that's
  a one-liner via `PngDecoder::is_apng()` before the
  `decode_static_with_limits` call. Out of scope here.

- **Consider tightening `sanitize_load_error` substring set.**
  (Cross-cutting, not blocking 4a.) The "too large" substring is
  shared with the workspace LocalFile arm's
  `"file is too large to preview"`. Both correctly map to the same
  user-facing string, so the categorization is robust. No action;
  noting it for awareness.
