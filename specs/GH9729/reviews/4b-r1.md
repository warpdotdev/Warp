# Findings

## Spec fidelity (exact)

- `MAX_ANIMATED_FRAMES = 256` matches §227. PASS.
- `MAX_ANIMATED_TOTAL_PIXELS = 67_108_864` matches §228 (8192 * 8192). PASS.
- Helper signature, body, and bail strings match §259's reference impl
  byte-for-byte (`tech.md` lines 269-310 vs. `image_cache.rs:316-374`):
  - `if i >= MAX_ANIMATED_FRAMES { bail }` BEFORE `let frame = frame?`.
  - `total_pixels = saturating_add(...); if total_pixels > MAX { bail }`
    BEFORE `frames.push(frame)`. PASS.
  - `if frames.is_empty() { bail("animated image has no frames") }`
    matches the spec exactly (it is prescribed in the spec block
    `tech.md:306-308`, not a defensive add by the implementer). PASS.
- Helper bails with `"decode_animated_with_limits called with non-animated
  format"` for unreachable formats; spec uses `unreachable!()`. The commit's
  graceful bail is strictly safer (no panic on programmer error) and not
  observable in legitimate flow. NIT, not a regression.

## Iteration / off-by-one

- `enumerate()` starts at `i = 0`. The cap check `i >= 256` bails on the
  257th call to `next()`. So 256 frames are accepted (i = 0..=255). This
  matches the natural reading of "MAX_ANIMATED_FRAMES = max accepted
  frames" and the spec text `if i >= MAX_ANIMATED_FRAMES { bail }`. PASS.
- Tiny-then-huge ladder: 255 1×1 frames + a 256th 8192×8192 frame leaves
  total at 255 + 67_108_864 = exactly `MAX_ANIMATED_TOTAL_PIXELS + 255`,
  which is `>` cap → bail on frame 256. Cap is `>`, not `>=`, so a
  frame-collection summing to exactly 67_108_864 succeeds. Matches §228
  ("total pixels across all frames" being a budget, not a strict-less-than).
  PASS.
- Saturating arithmetic on `u64` for both pixel-product (per frame) and
  running-sum: any saturation to `u64::MAX` makes the cap check trivially
  true, so saturation is a fail-safe behavior. PASS.

## Animated decoder verification (the load-bearing claim)

Verified against `image-0.25.9` source on this machine:

- `GifDecoder::set_limits` (`codecs/gif.rs:100-108`): calls
  `limits.check_support` and `limits.check_dimensions(self.dimensions())`,
  then **stores** `self.limits = limits`. The stored limits are cloned by
  `GifFrameIterator::new` (`codecs/gif.rs:248`) and `local_limits.
  reserve_buffer(width, height, COLOR_TYPE)` is called per frame
  (`codecs/gif.rs:325, 396`). So in this image version GIF DOES enforce
  `max_alloc` per frame — the commit body's claim that "GifDecoder ignores
  max_alloc per frame" is overly pessimistic for 0.25.9 (it likely came from
  an earlier `image` version). The explicit budget is still correct
  defense-in-depth and survives an `image` upgrade that drops the
  per-frame `reserve_buffer`. NIT only — the spec's pessimism is the
  safer prior to encode.
- `WebPDecoder` does **not** override `set_limits` (no match in
  `codecs/webp/decoder.rs`); inherits the trait default
  (`io/decoder.rs:115-120`) which calls `check_support` +
  `check_dimensions(self.dimensions())` and **discards** the rest — limits
  are not stored anywhere. `WebPDecoder::into_frames`
  (`codecs/webp/decoder.rs:103-145`) allocates `RgbaImage::new(width,
  height)` per frame using the canvas dimensions, no `reserve_buffer`.
  So WebP's animated `max_alloc` is genuinely unenforced — the trait
  default's `check_dimensions` only validates the canvas bounds (W ≤ 8192,
  H ≤ 8192) once at `set_limits` time. The commit's description is
  accurate. PASS.
- Net consequence: the dimension cap (canvas ≤ 8192 × 8192) is enforced
  for both formats at `set_limits` time. The frame-count cap and
  total-pixel cap are the only effective bound on the WebP animated
  envelope. The commit gets this right.

## Error mapping (UX nit, not blocking)

`app/src/workspace/lightbox_view.rs::sanitize_load_error` (lines 172-181)
matches on lower-cased `"too large" | "exceeds" | "decode" | "format"`:

- `"animated image has too many frames"` — matches none → falls through
  to `"could not read image"`. UX-suboptimal: this is a "too large" class
  error and should land in the "image is too large to preview" bucket.
- `"animated image exceeds total pixel budget"` — contains `"exceeds"` →
  `"image is too large to preview"`. CORRECT.
- `"animated image has no frames"` — matches none → `"could not read
  image"`. ACCEPTABLE (it really is a corrupt-input case, not a size
  case).

This is a **nit** for item 5b's `sanitize_load_error` table or for a
follow-up patch to either (a) add `"too many"` to the "too large" branch,
or (b) change the bail to `"animated image has too many frames (exceeds
frame budget)"` so the existing `"exceeds"` substring fires. The
correctness/security envelope is unaffected.

# Suggestions

1. (Nit, optional, post-merge.) Either widen `sanitize_load_error` to
   match `"too many"` or rephrase the bail to include `"exceeds"` so the
   existing rule fires. Currently the user-visible message for the
   frame-count cap will be `"could not read image"` (the truthful but
   weakest of the three lightbox messages) instead of `"image is too
   large to preview"`. Defer to item 5b.
2. (Nit only, not a defect.) The commit message claims `GifDecoder`
   ignores `max_alloc` per frame; in `image-0.25.9` it does enforce via
   `local_limits.reserve_buffer` per frame. The explicit cap is still
   correct as defense-in-depth for `image` upgrades, and the spec's
   pessimistic phrasing in §263 is the safer prior to encode. No code
   change needed; if the spec is ever revised, soften "does not enforce"
   to "may not enforce, depending on `image` version".
3. (Out of scope for 4b but worth flagging on 4c handoff.) `set_limits`'
  `check_dimensions` is called against `self.dimensions()` once. If a
  malformed file misreports `dimensions()` smaller than per-frame
  `read_frame` outputs, only the running total-pixel cap protects us.
  The current code reads `buf.width() * buf.height()` from the actual
  decoded `Frame`, so this is already handled correctly.
