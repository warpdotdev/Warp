# Findings

## Spec coverage (1c bullet)

The three tests prescribed by `tech.md` §613 (lines 617–619) are all present and correctly placed adjacent to the existing `test_resolve_file_target_*` cluster (insertion point line 333, well-grouped). Coverage matches the bullet wording 1:1:

- Sweep across `{png, jpg, jpeg, gif, webp, svg}` × both casings × all four `EditorChoice` variants × both `prefer_markdown_viewer` values: 12 × 4 × 2 = 96 cases. Comprehensive.
- SVG-vs-markdown precedence test is explicit (note: this case is also covered inside the sweep, so the dedicated test is technically redundant; the spec prescribes it separately, however, so the duplication is correct per §613).
- Non-image binary fall-through includes `bmp`, `tiff`, `ico` — i.e., the "deliberately out of v1 scope" raster/vector formats called out in `tech.md` §591–593. The comment in the test references this scope-call-out clearly. Good.

No prescribed fourth test in §613 was missed.

## Naming convention — primary nit

The three new tests drop the `test_` prefix: `resolve_file_target_image_preview_for_each_supported_extension`, etc. Every one of the 24 surrounding tests in the same `mod tests` block uses `test_` (verified across lines 257–625 of the file — see `grep -nE "^\s*fn"` results). This is the only `mod tests` in the file and it is uniformly prefixed. The new tests visibly break that convention without justification.

Rust does not require the `test_` prefix and Cargo style has been moving away from it, but local consistency wins inside a single test module. Recommend renaming to `test_resolve_file_target_image_preview_*` to match the surrounding cluster — particularly because two of the three new tests are direct extensions of the existing `test_resolve_file_target_*` family and would otherwise sort/group oddly in `cargo nextest list` output.

## Pre-existing test edits — `image.png` → `archive.zip`

Both edited tests (`test_resolve_file_target_binary_is_system_generic` and `test_resolve_file_target_binary_uses_env_editor`) remain meaningful after the swap. The first now actually tests what its name claims (a non-image binary routing to `SystemGeneric`); previously it was relying on `image.png` falling through `is_file_openable_in_warp` returning `None` because `is_supported_image_file` did not yet take precedence. With the §74 short-circuit landed, `image.png` now returns `ImagePreview`, which would have invalidated those tests. The leading comment on the first edit explains the reasoning and references §74; the second test cross-references the first via a one-line comment. Clean handoff. Not tautological.

## Idiomatic Rust

- `for ext in extensions` over `[T; N]` where `T: Copy` is idiomatic (Rust 2021 IntoIterator on arrays).
- `for editor_choice in editor_choices` works because `EditorChoice: Copy` (verified in `app/src/util/file/external_editor/settings.rs:7-16`). Same for `Editor` (`mod.rs:60-73`). No clippy concerns.
- `for prefer_markdown_viewer in [false, true]` is idiomatic and clearer than `for b in 0..2 { let prefer = b == 1; }`.
- `format!("photo.{ext}")` followed by `Path::new(&path_owned)` is clean — slightly verbose but more readable than `Path::new(&format!("photo.{ext}"))` inline (which would also work). The named binding makes the assertion message unambiguous.
- The assertion message `"ext={ext} editor_choice={editor_choice:?} prefer_markdown={prefer_markdown_viewer}"` correctly identifies the failing case in the 96-case product. Excellent.

## Test rigor

- Sweep test (test 1) has a precise per-case message — the failing iteration is fully recoverable from the panic output.
- Precedence test (test 2): single-shot, `assert_eq!` without a message — fine because there is only one case; failure mode is unambiguous.
- Non-image binary test (test 3) has `name={name}` — sufficient for the seven-element loop.

The asymmetric assertion-message density is appropriate, not a defect: parametrized loops need them; single-shot tests do not.

## Comment quality

All three test comments add real value rather than restating the function name:

- Test 1 explains *why* the cross-product matters (orthogonal flags).
- Test 2 explains *why* SVG specifically is the precedence canary (XML at the byte level, would otherwise route through markdown/code-editor). This is the strongest comment of the three.
- Test 3 explicitly calls out the v1-scope decision for `bmp/tiff/ico` and references `is_supported_image_file`.

The two pre-existing test edits also got accurate, useful comments.

## Determinism / brittleness

All three tests are deterministic. No filesystem touches (only `Path::new(...)`), no time, no env. `EditorChoice` and `Editor` derive `Eq`, so `assert_eq!` comparisons are exact. No flaky surface.

## Editor::VSCode under feature gates

Each new test is `#[cfg(feature = "local_fs")]` and references `EditorChoice::ExternalEditor(Editor::VSCode)`. `Editor::VSCode` is unconditionally compiled (`mod.rs:74-76`, no cfg gate). The pre-existing `test_resolve_file_target_markdown_viewer_precedence` also uses `Editor::VSCode` under the same `local_fs` gate, so this is established precedent. No problem.
