# Findings

- **Variant shape — idiomatic and spec-faithful.** `Error { message: String }` matches the named-field pattern shown in §182's literal Rust block (lines 187-192) and is consistent with the sibling `Resolved { asset_source: AssetSource }` form. A tuple `Error(String)` would have been technically simpler but inconsistent with the sibling and the spec; the named field also reads better at the call sites that destructure (`Error { message }` in `view.rs:23875`, `lightbox_view.rs:198/204/316/345`, `view_test.rs:3070/3090/3116`). No criticism here — this is the right call.

- **Naming — `Error` is fine in context, with one mild caveat.** §182 prescribes `Error`, so the name is locked. The Rust convention nit is real (the `Error` identifier shadows the `std::error::Error` trait inside any module that brings the variant into unqualified scope), but every actual call site I read uses fully-qualified paths (`lightbox::LightboxImageSource::Error` or `LightboxImageSource::Error`), so the shadow risk is theoretical, not actual. Not worth pushing back on.

- **Doc-comment style — slightly heavier than siblings, but justified.** `Loading` and `Resolved` get one-line and two-line doc comments respectively. The new `Error` variant gets ~8 lines split into two paragraphs (behavior summary, then a security note about sanitization, with a `tech.md §182` xref). This is heavier than the siblings, but the imbalance is appropriate: `Error` carries a security invariant (do NOT interpolate raw OS errors / paths) that the siblings do not. Encoding that invariant on the type itself, where every future caller will see it, is exactly where it belongs — much better than relying on §179 of the spec or a code-review catch. Style nit: the second paragraph could be tightened (e.g., drop "human-readable" — `String` already implies that), but the content earns the lines.

- **Match-arm comment redundancy — one of the two consumer-site comments is the load-bearing one; the other is restating the obvious.** The `lightbox_view.rs:165` comment ("Error entries have no native size; the existing render logic tolerates `None` here. See specs/GH9729/tech.md §182.") is the one worth keeping — it cross-refs the spec and explains the `None` choice. The `library.rs:740` comment ("Error entries have no native size.") is redundant with both the arm body and the sibling `Loading => None` arm just above it (which has no comment). I would elide the example-binary comment entirely, since the sibling `Loading` arm sets the precedent that "no native size" is self-evident from `=> None`. Not blocking — examples are allowed slightly more verbosity — but flagging for consistency.

- **Match-arm comment placement vs. sibling `Loading`.** The sibling `Loading => None,` arm has no comment, and the new `Error { .. } => None,` arm gets one. Within the same match block, asymmetric commenting reads as "this case is special" — and it isn't, mechanically. The asymmetry is fine if the comment is doing real cross-referencing work (which the lightbox_view.rs version is), but I'd argue the comment belongs on the function-level doc-comment of `current_image_native_size` (one comment covering both `Loading` and `Error` returning `None`), not on the `Error` arm alone.

- **Module boundaries — correct.** `LightboxImageSource` lives in `crates/ui_components/src/lightbox.rs` (the UI component that owns rendering); the workspace layer in `app/src/workspace/` constructs values of it (`view.rs:23875` for the metadata-stat path, `lightbox_view.rs:198/204` for the post-load callback path). This is the right direction: UI primitive owns the type, workspace adapts loading state into it. No leakage.

- **Test rigor — type-only change with no new tests is the right split.** 3a only adds an enum variant and two `=> None` arms; there is nothing behavioral to test at this commit. The render path is added in 3b (`328c333`), and the post-load callback that actually constructs `Error` values from `FailedToLoad` / `Unrecognized` is added later in `d5fdacc` ("post-load callback rewrites failures to Error variant"). The acceptance tests in `view_test.rs:3070-3120` exercise the full chain. So the absence-of-tests in 3a is justified by the splitting strategy — but see next point.

- **3b also has no rendering tests.** `git show --stat 328c333` shows `crates/ui_components/src/lightbox.rs | 23 +++` and only the TODO file alongside — no test additions. The render arm constructs a `Flex::column()` with two `Text` children and is non-trivial enough to merit at least a smoke test (e.g., render a `LightboxImageSource::Error` at the integration layer and assert no panic / correct text). The workspace-side `view_test.rs` cases at 3070+ test the *construction* of `Error` from various failure modes, but they do not exercise the `lightbox.rs` render arm. This is a 3b finding, not a 3a one — but it is the natural follow-on to the 3a "no tests is fine because 3b adds them" claim, and 3b doesn't fully discharge it. Worth flagging when 3b is reviewed.

- **Dead-code window between 3a and 3b is acceptable.** Between c0b5a7c and 328c333 (one commit later, ~2 minutes apart per the timestamps), the `Error` variant exists but the `Lightbox::render` match falls through to its catch-all and renders nothing distinct for it. This is a tolerable interim state — `cargo check` is clean, and the catch-all already handled "anything we don't know about" before this commit. A single atomic 3a+3b commit would have been defensible, but the split reads cleanly for review (type-shape change vs. render-behavior change), and the commit message in c0b5a7c explicitly anticipates and explains the gap. Fine as-is.

- **Two truly-exhaustive consumer sites, both updated.** The commit message correctly identifies that only `current_image_native_size` (lightbox_view.rs:154) and `native_size_for_image` (library.rs:729) match exhaustively on `LightboxImageSource`; everywhere else uses `if let Resolved { .. }` or has a catch-all. I cross-checked with `grep -rn "LightboxImageSource::"` and confirmed: no other exhaustive match exists in the tree. The commit message is accurate.

- **`#[derive(Clone, Debug)]` carries through correctly.** The new `Error { message: String }` payload is `Clone + Debug`, so the existing derive on the enum keeps working. No `PartialEq` is derived — looking at `view_test.rs:3070+`, the tests destructure with `if let LightboxImageSource::Error { message } = ... ` rather than asserting equality, which sidesteps the lack of `PartialEq`. Acceptable; adding `PartialEq` would be a separate change.
