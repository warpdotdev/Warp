# Findings

- **Lightbox sanitizer category mismatch (concern).** `app/src/workspace/lightbox_view.rs:172-180`
  buckets errors by lowercased substring: `"too large"` or `"exceeds"` →
  `"image is too large to preview"`; `"decode"` or `"format"` →
  `"could not decode image"`; otherwise `"could not read image"`. The new
  message in this commit is `"svg dimensions exceed render budget"`. Neither
  `"too large"` nor `"exceeds"` is a substring of that message — it has
  `"exceed "` (verb form, no trailing `s`). So when an over-cap SVG trips
  this gate, the user sees `"could not read image"` rather than
  `"image is too large to preview"`. Compare to the sibling decoder messages
  that route correctly: static raster `"image is too large to preview"`
  (matches `"too large"`) and animated `"animated image exceeds total pixel
  budget"` (matches `"exceeds"`). The two SVG-branch bails are the only
  decoder-side reject messages in this PR series that miss the sanitizer
  buckets. This isn't a security defect — the error is still surfaced — but
  the user-facing categorization is wrong, and "could not read image" is
  misleading for what is actually a too-large case. Three fixes available,
  in order of preference: (1) change the message to include `"exceeds"`
  (e.g. `"svg dimensions exceed[s] render budget"`); (2) change it to
  include `"too large"` (e.g. `"svg too large to preview"`); (3) extend
  `sanitize_load_error` to also match `"exceed"` (riskier, casts a wider net
  for any future error string with "exceed" in a non-size sense).

- **Identical message for two distinct bail conditions (nit).** Both the
  per-axis check (`w > MAX_SVG_RENDER_DIMENSION || h > MAX_SVG_RENDER_DIMENSION`)
  and the pixel-product check (`pixels > MAX_DECODE_PIXELS`) emit the same
  string `"svg dimensions exceed render budget"`. From a user-facing
  standpoint this is fine (both mean "image too large"), but for operator
  log-grepping it costs the ability to tell which gate fired. Differentiating
  (e.g. `"svg dimensions exceed render budget"` vs
  `"svg pixel product exceeds budget"`) would also fix the sanitizer-bucket
  issue above for the second branch by construction, since the second
  message would contain `"exceeds"`.

- **`f32 as u32` truncation semantics (nit, accepted).** `usvg::Tree::size()`
  returns `tiny_skia_path::Size`, whose `width()`/`height()` are `f32`. The
  `as u32` cast truncates toward zero (and saturates at `u32::MAX` for
  out-of-range values, with NaN → 0, negative → 0 — all safe). For the
  attack case (`width="200000"`), 200000.0 as u32 = 200000, cap fires,
  good. For pathological fractional cases, `width="0.5"` truncates to 0
  and passes the cap; that's acceptable degradation since 0.5 px is
  trivially under any rasterizer concern. For width="8192.7" the cast
  yields 8192 and passes the per-axis cap, but resvg would render as if
  ceil()'d to 8193; the error here is one pixel and well under the 64M
  pixel-product cap. Using `ceil()` first would be slightly more
  conservative but is cosmetic. Spec §321 example uses the same `as u32`
  cast, so the implementation matches the spec verbatim.

- **`pub(crate)` visibility justified (pass).** `looks_like_svg_xml` is
  `pub(crate)` so `crates/warpui_core/src/assets/asset_cache.rs` (item 5a)
  can reuse it without re-implementing the predicate. The doc comment
  explicitly enumerates both callers and the rationale for sharing the
  predicate (cap signal and parser-gate signal aligned by construction).
  This is the right shape; resisting `pub` keeps it internal to the crate.

- **`is_ascii_whitespace` matches XML 1.0 S production (pass).** Rust's
  `u8::is_ascii_whitespace` matches U+0009 (HT), U+000A (LF), U+000C (FF),
  U+000D (CR), U+0020 (SP). XML 1.0 §2.3 S production is exactly
  `(#x20 | #x9 | #xD | #xA)` — a strict subset. The Rust predicate is
  one byte (U+000C, FF) wider than the XML S production, which is
  intentional defensive looseness for the prefix-sniff role.

- **Bounded scan (pass).** `bytes.len().min(1024)` correctly caps the
  whitespace scan at 1 KB to refuse a "1 GB of leading whitespace" DoS
  input. The comment calls this out explicitly.

- **Doc comment density (pass).** 16-line doc comment on
  `looks_like_svg_xml` is unusually heavy for a one-screen helper, but for
  a security predicate with two callers that's warranted. The comment
  explicitly names both callers, the rationale for case-sensitivity, and
  the "coarse sniff, not a full lexer" boundary — matches the spec text.

- **Idiomatic Rust (pass).** `data.strip_prefix(&[0xEF, 0xBB, 0xBF]).unwrap_or(data)`
  is idiomatic for optional BOM strip. `bytes[..scan_end].iter().position(…)
  .map(|i| &bytes[i..]).unwrap_or(&[])` is a clear chained iterator. Four
  `starts_with` chained with `||` is verbose but grep-friendly; a prefix
  array + `.iter().any(|p| after_ws.starts_with(p))` would be terser at
  the cost of literal-search discoverability — leaving as-is is the right
  call.

- **Constant placement and naming (pass).** `MAX_SVG_RENDER_DIMENSION` is
  grouped with the other GH9729 SVG/decode constants
  (`MAX_DECODE_DIMENSION`, `MAX_DECODE_ALLOC`, `MAX_ANIMATED_FRAMES`,
  `MAX_ANIMATED_TOTAL_PIXELS`). Name parallels `MAX_DECODE_DIMENSION`.
  Function `looks_like_svg_xml` is descriptive and unambiguous.

- **Refactor of the SVG branch (pass).** Original was 5 lines (sniff
  `data.first() == Some(&b'<')`, parse, wrap in `Rc`); new version is
  18 lines with sniff, parse, two checks, wrap. The growth is justified
  by §321 (sanity gate + intrinsic cap). Inline comments are appropriately
  distributed: one comment block on the sniff/predicate-rationale, one on
  the §321 intrinsic-dimension cap. Higher comment density than the JPEG/
  PNG branches matches the higher security weight.

- **`saturating_mul` on `(u32 as u64)` (pass).** Width and height are each
  ≤ u32::MAX after the prior bound, but the spec's pixel-product check
  uses `saturating_mul` defensively in case the intrinsic cap is later
  loosened. Idiomatic.

- **Test coverage (pass — out of scope here).** This commit ships no
  tests; tests live in 4-tests-a (`abab04227375`). Confirmed via
  `git show abab042 --stat`: SVG cases covered are (1) non-xml prefix
  rejection (`decode_svg_rejects_non_xml_prefix`), (2) BOM + whitespace +
  `<?xml` acceptance, (3) `<!DOCTYPE` acceptance, (4) `<!--` comment
  prelude acceptance, (5) normal 256x256 icon round-trip
  (`decode_svg_accepts_normal_icon`), (6) 200000x200000 intrinsic-
  dimension cap (`decode_svg_rejects_intrinsic_dimensions_over_cap`).
  Coverage matches §613 (lines 647-652).

# Suggestions

1. **(Concern, recommended fix in this PR or a follow-up commit before
   item 4-tests-b lands.)** Change the SVG bail strings so the lightbox
   sanitizer routes them to `"image is too large to preview"`. Two
   options:

   ```rust
   if w > MAX_SVG_RENDER_DIMENSION || h > MAX_SVG_RENDER_DIMENSION {
       anyhow::bail!("svg dimensions exceeds render budget");
   }
   let pixels = (w as u64).saturating_mul(h as u64);
   if pixels > MAX_DECODE_PIXELS {
       anyhow::bail!("svg pixel product exceeds render budget");
   }
   ```

   Both messages now contain `"exceeds"` and route to the correct
   user-facing category. Differentiating the two bails also gives
   operator log-grep the ability to distinguish per-axis cap fires from
   pixel-product cap fires. (Spec §321 line 370 uses
   `"svg dimensions exceed render budget"` literally, so the spec
   wording also wants a one-character touch-up; flagging for the spec
   doc as well.)

2. **(Nit, optional.)** If the code stays with the current message, add
   a regression test in `lightbox_view.rs` (alongside the existing
   `sanitize_load_error_picks_too_large_category` test) that constructs
   an `anyhow!("svg dimensions exceed render budget")` and asserts it
   maps to `"image is too large to preview"`. This would have caught the
   mismatch and locks in the contract going forward regardless of which
   side gets edited.

3. **(Nit, optional.)** `const BOM: &[u8] = &[0xEF, 0xBB, 0xBF];` at
   module scope, then `data.strip_prefix(BOM).unwrap_or(data)` — slight
   readability win. Trade-off is one more named constant in scope; not
   worth a churn edit on its own.
