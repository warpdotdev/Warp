# Findings

- **Test naming follows the spec, not the file convention.** The 14 pre-existing tests in `image_cache_tests.rs` (lines 62-565) all use a `test_` prefix (`test_passes_through_asset_cache_original`, `test_svg_text_rasterizes_with_loaded_system_fonts`, etc.). The 9 new tests use a `decode_` prefix. This is a deviation from the surrounding convention, but it is **driven by the spec** — `tech.md` §613:640-652 explicitly names every test `decode_static_*` / `decode_svg_*`. The implementation matched the spec verbatim. Net: nit-level drift, justified by spec-conformance. If anything, the spec itself silently introduced the naming change. Worth a one-line comment in the test section header pointing back to §613:640 so future readers don't think it's a stylistic accident.
- **Module wiring confirmed:** `image_cache.rs:1167-1169` declares `#[cfg(test)] #[path = "image_cache_tests.rs"] mod tests;`. This makes the sibling file a child module of `image_cache`, so `super::decode_static_with_limits_inner`, `super::looks_like_svg_xml`, `super::MAX_DECODE_PIXELS`, and `super::ImageType` are all reachable through child-module privacy without any `pub(crate)` widening of the production surface. ✓ Correct pattern; no API leak.
- **Production refactor inside a "tests" commit.** This commit splits `decode_static_with_limits` into a thin wrapper plus a parameterized `decode_static_with_limits_inner(data, format, limits, max_pixels)`. That is a *production* code change inside a commit titled "decoder tests" — the same shape R2 has flagged on prior `*-tests` commits. The trade-off here is real: the spec requires testing rejection paths (`decode_static_rejects_dimensions_over_cap`, `decode_static_rejects_pixels_over_cap`) against modest fixtures, and the only clean way to do that without materializing 8192-px PNGs is to thread limits as parameters. The refactor is mechanical, the wrapper preserves production behavior identically (`decode_static_with_limits_inner(data, format, decode_limits(), MAX_DECODE_PIXELS)`), and the diff is small enough to review in-place. Acceptable, but the commit message should call it out more prominently than the current "Refactor:" paragraph (it does mention it — adequate).
- **Spec-vs-implementation fixture deviation.** Spec lines 640-642 prescribe synthesizing PNG headers declaring `20000×20000` and `9000×8000` against the production constants. The implementation instead uses a 200×100 PNG against shrunk caps (`max_image_width=100` and `max_pixels=10_000`). This achieves the same *intent* — exercising both the per-axis cap and the post-decode pixel cap — and is arguably more robust (the production constants can change without the test breaking). But it is a real deviation from the spec text. The reasoning is sound (avoid materializing 8192-pixel PNGs) and the commit message owns the trade-off. Recommend a short forward-link in `tech.md` §613:640-642 noting the inner-helper testing strategy, or accept the spec drift as documented in the commit message. Either is fine; the test rigor is preserved.
- **Brittleness against `image-rs` cap-enforcement order.** `decode_static_rejects_dimensions_over_cap` relies on `image-rs` enforcing `max_image_width`/`max_image_height` *before* materializing pixels. `decode_static_rejects_pixels_over_cap` uses dimensions safely under both per-axis caps so the post-decode `MAX_DECODE_PIXELS` branch is the only one that can fire. The two tests cleanly partition the two failure modes. ✓ Robust against `image-rs` reordering: even if `image-rs` started enforcing pixels-before-dimensions (or vice versa), each test would still fail for the right reason because the *opposite* cap is set wide enough to be a no-op. Non-issue.
- **`encode_blank_png` helper is idiomatic.** `image::DynamicImage::ImageRgba8(img).write_to(&mut Cursor::new(&mut out), ImageFormat::Png).expect(...)` is the canonical `image` crate pattern for in-memory PNG encoding. The helper is small, deterministic, and the `expect` message is informative. ✓
- **`assert!(matches!(...))` vs `match { _ => panic!(...) }`.** `decode_svg_accepts_normal_icon` uses an explicit `match` instead of `assert!(matches!(result, super::ImageType::Svg { .. }))`. The explicit form is verbose, but `panic!("expected Svg variant")` gives a slightly nicer Display than `assert!(matches!)` (which prints the full `Debug` of the value if the variant differs). Trade-off; no change required. The other `assert!(result.is_err(), "expected ...")` calls all carry custom messages — good rigor.
- **Predicate vs end-to-end split is sensible.** Three SVG predicates exercise `looks_like_svg_xml` directly (NUL-prefix reject, BOM/whitespace accept, DOCTYPE accept, comment accept). Two SVG end-to-ends (`decode_svg_accepts_normal_icon`, `decode_svg_rejects_intrinsic_dimensions_over_cap`) go through `ImageType::try_from_bytes`. This is the right partitioning: predicate tests cover the gate's input space cheaply, end-to-end tests cover the gate-to-rasterizer-to-cap chain. ✓
- **Coverage gaps relative to GH9729 surface area** (these are acceptable for §613:640-652 but worth flagging for completeness):
  - **No JPEG accept test.** `decode_static_accepts_normal_photo` only covers PNG. The static-decode path also handles JPEG and WebP-static. If `image-rs` `Limits` enforcement diverges across codecs, the test wouldn't catch it.
  - **No WebP-static accept test.**
  - **No mixed-case SVG predicate fail test** (e.g. `<SvG`) — confirms the gate is case-sensitive (or, if it isn't, that `usvg` accepts).
  - **No empty-buffer predicate test** (`looks_like_svg_xml(&[])`). Trivial to add and cheap insurance against an off-by-one in the prefix check.
  - **No end-to-end BOM-SVG test** — predicate-only coverage. `decode_svg_accepts_xml_prelude_with_bom_and_whitespace` runs through `looks_like_svg_xml` but never confirms `usvg::Tree::from_data` actually accepts the BOM-prefixed payload end-to-end.
- **Animated tests deferred to 4-tests-b.** Documented in the commit message and reflected in `IMPLEMENTATION_TODO.md`. The split honors the loop rule (one bullet = one commit) and the rationale (animated WebP fixture synthesis is non-trivial) is sound. ✓
- **Fixture sizing aligned with spec.** All fixtures are small (200×100 PNG, 256×256 SVG, declared-only 200000×200000 SVG byte payload, 1 KB NUL buffer). No test materializes a multi-megabyte payload. ✓ aligned with the §613 emphasis on modest fixtures.
- **No test relies on `image-rs` decode-then-check order silently passing for the wrong reason.** The two static rejection tests partition the failure modes (see "Brittleness" finding above). ✓
