# Findings

## Spec fidelity (matches §321)

- `MAX_SVG_RENDER_DIMENSION = 8_192` — present, matches §321.
- `looks_like_svg_xml` predicate body is byte-for-byte identical to the snippet in §321 (BOM strip, 1 KB whitespace bound, four start tokens `<?xml | <svg | <!-- | <!DOCTYPE`). Token list, ordering, case-sensitivity all match.
- Pre-parse content-sanity gate: §321 prescribes `if !looks_like_svg_xml(data) { bail }` before `usvg::Tree::from_data`. Implementation uses `if looks_like_svg_xml(data) { … usvg::Tree::from_data … } else { fall through to image::guess_format }` which is the same gate in inverted form: a non-XML blob never reaches `usvg`. Equivalent and correct.
- Intrinsic-dimension check fires AFTER `usvg::Tree::from_data` and BEFORE rasterization (rasterization happens later in `RenderImage::paint` / `resvg::render`, not in `try_from_bytes`). Layering matches §321.
- Pixel-product check uses `MAX_DECODE_PIXELS` (the 67M constant from item 4a). §321 names `MAX_DECODE_PIXELS` directly in its code snippet, so this is spec-prescribed. Sharing is appropriate: SVG rasterization produces an RGBA bitmap of `w * h` pixels just like a static raster decode, so the same per-image pixel ceiling applies.
- The `pub(crate) looks_like_svg_xml` is reachable from item 5a — confirmed at `crates/warpui_core/src/assets/asset_cache.rs:542` (`crate::image_cache::looks_like_svg_xml(&peek[..n])`). Cap signal and parser-gate signal are aligned by construction.

## Predicate edge cases (all behave correctly)

- Empty input: `bytes=[]`, `scan_end=0`, `position` returns `None`, `after_ws=&[]`, `starts_with` on non-empty prefix is false. Rejected. ✓
- Pure whitespace: same path as empty. Rejected. ✓
- Just BOM: post-strip is empty. Rejected. ✓
- BOM + whitespace + `<svg`: BOM stripped, position finds `<` inside the 1 KB window, `after_ws` starts with `<svg`. Accepted. ✓
- Pathological `> 1024` bytes of leading whitespace then `<svg`: `position(...)` returns `None` over the bounded window, `after_ws=&[]`. Rejected. Acceptable — tightly aligns with the spec's stated intent of refusing pathological inputs.
- Mixed-case `<SVG`: rejected per spec choice (case-sensitive). usvg would reject it too (XML is case-sensitive); divergence here is harmless.
- The four `starts_with` checks each compare slices against fixed byte literals; `<!-- ` and `<!DOCTYPE` order does not interact with `<!`-prefixed shorter token because there is no `<!`-only token in the list. Disjoint, correct.

## SVG attack-class audit

1. **Billion-laughs / nested entity expansion** — `usvg` 0.47 calls `roxmltree::Document::parse_with_options(text, {allow_dtd: true, …})`. roxmltree itself ships a `LoopDetector` (`parse.rs:470`): max entity depth 10, max 255 references per root reference. Billion-laughs is mitigated **by roxmltree**, not by usvg. The commit body's claim that "usvg disables external-entity expansion by default" is more nuanced: usvg sets no entity-expansion flag of its own; the protection is inherited from roxmltree. **Defence is real, but it lives one layer down from where the commit body locates it.** A future usvg or roxmltree bump could change this without the warp side noticing. Worth a CI dep-pin or a comment.
2. **External entities (XXE)** — roxmltree exposes `entity_resolver: Option<&EntityResolver>` (default `None`). usvg's `from_str` does NOT set an `entity_resolver`, so external entity references are simply discarded by roxmltree. No file disclosure via `<!ENTITY foo SYSTEM "file:///etc/passwd">`. Correct — but again, the protection is inherited and not asserted explicitly in our code. If usvg ever started passing an entity_resolver, we'd lose it silently.
3. **Tiny-byte intrinsic-dimension bomb** — caught by the new dimension cap before any pixel allocation. Verified end-to-end via `decode_svg_rejects_intrinsic_dimensions_over_cap` in `image_cache_tests.rs:678-689`. ✓
4. **`<script>` in SVG** — usvg's tree model has no script node, scripts are dropped at parse time. No JS engine in `warpui_core` regardless. Not a vector. ✓
5. **Font lookup** — `fontdb: SVG_FONT_DB.clone()` pre-populates the font db; `usvg::FontResolver::default()` resolves from the supplied db only, no network. ✓
6. **`<image href="https://...">` external HTTP** — `ImageHrefResolver::default_string_resolver` (parser/image.rs:85-114) treats `href` as a **filesystem path**, not a URL. URLs reach `path.exists()` which returns false, and the image is skipped. No network IO. ✓
7. **`<image href="/etc/passwd"> ` LOCAL file disclosure (CONCERN — see below)** — `default_string_resolver` calls `std::fs::read(&path)` on whatever `<image href="...">` contains, then attempts to interpret the bytes as PNG/JPEG/GIF/WebP/SVG. The doc on `ImageHrefResolver` itself flags this: *"you can forbid access to local files (which is allowed by default)."* Two implications:
   - **File-existence probe oracle.** A malicious SVG can include `<image href="/Users/$USER/.ssh/id_rsa"/>` — the file is opened and read, but only succeeds in producing pixels if the bytes happen to be a valid image. So it isn't a direct content-exfiltration vector (no JS in the SVG to read pixels back over the network), but **the load DOES happen** and consumes IO bandwidth. A 10 GB file referenced via `<image href="/path/to/huge.bin"/>` is read fully into memory by `std::fs::read` before usvg discards it as not-an-image. This bypasses the asset-cache's 4 MB SVG byte cap entirely — that cap protects the SVG bytes themselves, not files referenced from inside.
   - **TOCTOU on referenced-file-shaped-as-image.** If the local file IS a valid image (e.g. user has `~/Pictures/foo.png`), an SVG renders that image as part of its output. Net effect on the warp file-tree image-preview path: a malicious `.svg` could embed contents of arbitrary user-readable files into a rendered preview. Low severity in v1 (preview only renders to the local terminal, not exfiltrated), but the threat surface is real and not mentioned in §321 or in the commit body.
   - **Mitigation options** to consider: (a) override `image_href_resolver.resolve_string` with `Box::new(|_,_| None)` — same treatment usvg itself applies in `from_data_nested` for sub-SVGs (see `parser/mod.rs:122-126`); (b) override `resolve_data` to bound the size of base64 payloads (today there is no cap on the inline raster size that an SVG can carry, see attack class 8).
8. **Embedded base64 raster inside SVG** — `<image href="data:image/png;base64,..."/>` is decoded by `ImageHrefResolver::default_data_resolver` and the raw `Arc<Vec<u8>>` is handed to `ImageKind::PNG(...)`. **The intrinsic-dimension cap on the SVG root does NOT bound the embedded raster's dimensions or byte size.** A 4 MB SVG with `<svg width="1" height="1"><image x="0" y="0" width="1" height="1" href="data:image/png;base64,...100000x100000-PNG..."/></svg>` passes the dimension cap (1×1) but at render time `resvg` will decompose-and-rasterize the 100000×100000 nested PNG. The 4 MB pre-parse byte cap (item 5a) does limit the base64-encoded bytes inside the SVG to ~3 MB of decoded raster, so this is bounded by file-size at the asset-cache layer, but the dimension cap and `MAX_DECODE_PIXELS` we just added do NOT see the embedded raster's intrinsic dimensions. **Acceptable for v1** because the 4 MB cap is the binding constraint, but worth calling out in §321 since the spec presents the dimension cap as "the" defense against the dimension-bomb class.
9. **UTF-16 SVG** — predicate strips only UTF-8 BOM; UTF-16 (`0xFF 0xFE` / `0xFE 0xFF`) input is rejected. usvg's `from_data` itself only accepts UTF-8 (`std::str::from_utf8(data)?` at `parser/mod.rs:104`), so degradation is identical: rejected before rasterization. Net behavior is consistent — a UTF-16 SVG produces "could not decode" both before and after this commit. ✓
10. **SVGZ (gzip-compressed SVG)** — usvg's `from_data` accepts SVGZ via the `0x1f 0x8b` magic at `parser/mod.rs:99`. Our `looks_like_svg_xml` predicate REJECTS gzip-prefixed bytes (the gzip magic is binary, not whitespace or `<`). So the `try_from_bytes` route now refuses SVGZ files that previously may have parsed via the loose `data.first() == Some(&b'<')` heuristic (which also failed on SVGZ — gzip first byte is 0x1f, not `<` — so this is **NOT a regression**). Net: SVGZ was unsupported before this commit and is unsupported after. Worth confirming with item 5a/owners that this is intentional; if SVGZ support is expected, the predicate would need a `0x1f 0x8b` branch (and the bytes would still need to fit under `MAX_SVG_BYTES` post-decompression — itself a separate decode-bomb concern).
11. **Saturating cast `f32 as u32`** — `Size` from tiny-skia-path 0.12 guarantees `width()`/`height()` are positive non-zero finite f32 (per `NonZeroPositiveF32`, `tiny-skia-path-0.12.0/src/size.rs:128-135`). Cast is well-defined: any positive f32 ≤ u32::MAX casts losslessly modulo truncation; values > u32::MAX saturate to `u32::MAX`. Truncation is lenient by < 1 px per axis vs tiny-skia's `to_int_size` (which uses `round`). At cap-edge this means an SVG declaring `width=8192.9 height=8192.9` casts to 8192×8192 (passes cap) but the renderer's `to_int_size` rounds to 8193×8193 (~0.02% over `MAX_DECODE_PIXELS`). Negligible, not an attack vector, but worth a comment if we want exact alignment.
12. **CSS-pixel vs raster-pixel confusion** — `MAX_SVG_RENDER_DIMENSION = 8192` is the SVG's intrinsic CSS-pixel size. The actual rasterization may apply a DPI scale or a fit-to-target transform. For the file-tree preview path the target raster is bounded (the preview pane is small), so this is fine. For `RenderImage::paint` arbitrary scale-up, the dimension cap bounds the SVG's *intrinsic* size, not the *rendered* size — a 1×1 SVG scaled to 100k×100k at paint time would not be caught here. §321 implicitly accepts this; the rendering-pipeline-side budget is out of scope for 4c.

Coverage gaps (would be helpful but not blocking):

# Suggestions

1. **(non-blocking, recommend addressing in 4c follow-up or 5a)** Override `image_href_resolver.resolve_string` to `Box::new(|_, _| None)` — i.e. apply the same treatment usvg uses for nested SVGs (`parser/mod.rs:122-126`) at the top level. Closes attack class 7 (arbitrary-local-file read via `<image href="/path"/>`). Single-line change at the SVG parse site:

    ```rust
    let options = usvg::Options {
        fontdb: SVG_FONT_DB.clone(),
        image_href_resolver: usvg::ImageHrefResolver {
            resolve_data: usvg::ImageHrefResolver::default_data_resolver(),
            resolve_string: Box::new(|_, _| None),
        },
        ..Default::default()
    };
    ```

2. **(documentation, non-blocking)** Update §321 to clarify that the billion-laughs and XXE protections are inherited from `roxmltree`'s `LoopDetector` and from usvg leaving `entity_resolver = None`, not from any explicit usvg flag. Pin or assert the dependency versions where this protection lives, or add a comment near the parse site flagging this as a watch-on-version-bump invariant.

3. **(documentation, non-blocking)** Note in §321 that the dimension cap on the SVG root does NOT bound nested embedded rasters (`<image href="data:..."/>`); the byte cap from item 5 is the binding constraint for that attack class.

4. **(test coverage, non-blocking)** Add tests for: (a) > 1 KB leading whitespace rejection; (b) SVG missing intrinsic dimensions hits the usvg 100×100 default and accepts; (c) regression guard that SVGZ is still routed to the catch-all path. Optionally a test asserting `<image href="/dev/null"/>` doesn't crash and produces a sensible result, to lock in the local-file-load surface behavior either before or after suggestion 1 lands.

5. **(nit)** Consider rounding `size.width()` rather than truncating, to match tiny-skia's `to_int_size` semantics — eliminates the < 1 px lenience at cap edge. Cosmetic.
