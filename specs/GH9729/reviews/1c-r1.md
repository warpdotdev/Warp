# Findings

- **Spec fidelity is solid.** All three tests prescribed in tech.md §613 are present and named almost verbatim:
  - `resolve_file_target_image_preview_for_each_supported_extension` — sweeps `png/jpg/jpeg/gif/webp/svg` in lower and upper case (12 strings), four `EditorChoice` values, and both `prefer_markdown_viewer` settings. The spec asks for "regardless of `prefer_markdown_viewer` and `editor_choice`"; the cross-product (96 calls) covers that thoroughly.
  - `resolve_file_target_image_preview_takes_precedence_over_markdown` — `.svg` with `prefer_markdown_viewer = true` and `EditorChoice::Warp`. Matches the spec assertion.
  - `resolve_file_target_non_image_binary_still_system_generic` — covers all four spec-named binaries (`.zip`, `.mp3`, `.exe`, `.pdf`) plus the three explicitly-out-of-v1-scope formats `.bmp`, `.tiff`, `.ico`. The latter three go beyond the spec but are aligned with the §613 list and provide a useful guardrail against accidentally widening `is_supported_image_file`. Acceptable beyond-spec coverage.

- **Resolver behavior is correctly exercised.** `is_supported_image_file` (lines 76-87) uses `Path::extension()` then `to_ascii_lowercase()`; the resolver's step-0 short-circuit at line 212 calls it before any other classification. Mentally executing the inner loop: `format!("photo.{ext}")` → `Path::new(...)` → resolver step 0 → `is_supported_image_file` → `extension() = Some("png"|"PNG"|...)` → `to_ascii_lowercase()` matches → returns `FileTarget::ImagePreview` → assertion passes. Behavior matches the test name and spec line.

- **Determinism.** The tests are pure: no randomness, no env reads, no filesystem touches. `Path::new("photo.png")` is constructed from a literal `&str`, `EditorChoice` and `EditorLayout` are both `Copy`, and the resolver's step-0 path never reaches `is_file_openable_in_warp` (which is also lexical). The 96-call sweep is repeatable.

- **No filesystem side effects.** Confirmed by reading the resolver flow. `is_supported_image_file` only inspects the path's extension via `Path::extension()`, which is a pure string operation. No `std::fs` calls, no subprocesses.

- **The two existing-test renames are correct.** `test_resolve_file_target_binary_is_system_generic` and `test_resolve_file_target_binary_uses_env_editor` previously used `image.png`. With change 1b's short-circuit, `image.png` now routes to `ImagePreview`, so the original assertions (`SystemGeneric` and `EnvEditor`) would fail. Renaming the sample to `archive.zip` preserves the original intent (testing the binary fall-through). The diff also added inline comments explaining the rename, which is the right artifact for future readers. Note: the previous tests passed because the resolver did not yet have step 0; they were testing the binary path correctly under the pre-1b semantics.

- **Security surface is nil.** Pure unit tests on path strings; no disk writes, no subprocesses, no network.

# Suggestions

These are nits, not blockers. The verdict is pass-with-nits because the tests fully cover what §613 prescribes; the items below are optional hardening:

- **Mixed-case (e.g. `Photo.PnG`).** The diff sweeps fully-lower and fully-upper but not mixed casings. `is_supported_image_file` calls `to_ascii_lowercase()` on the extension, so `PnG` would also match. Adding one or two mixed-case strings (`"PnG"`, `"jPg"`) to the extensions array is a one-line change that future-proofs against someone replacing the lowercase-then-match with a hand-rolled `==` check. Not spec-required.

- **Path with directory components (e.g. `dir/sub/photo.png`).** All test paths are bare filenames. `Path::extension()` is documented to look only at the final component, so this should work, but a single sanity assertion (`Path::new("a/b/c/photo.png")` resolves to `ImagePreview`) would lock that contract. Optional.

- **Path with multiple dots (e.g. `backup.png.bak`).** Extension is `bak`; should NOT route to `ImagePreview`. The negative test's `archive.zip` style does cover the "extension is not an image" axis, but `backup.png.bak` specifically defends against a regression where someone replaces `Path::extension()` with a substring-match-on-name. Adding it to the negative list is a one-character addition and worth it. Optional.

- **No-extension and dotfile (e.g. `README`, `.hiddenfile`).** `Path::extension()` returns `None` for `README` and `Some("hiddenfile")` for `.hiddenfile` (Rust treats leading-dot files as having an extension equal to the name-after-dot). Neither matches an image extension, so both fall through. A single negative assertion for each would be a useful smoke test of the `unwrap_or(false)` branch on line 86. Optional.

- **The 96-call combinatorial sweep is fine but has zero variance on `default_layout` and `layout`.** Both are `EditorLayout::SplitPane` and `None`. The spec doesn't ask the resolver to be invariant on layout for ImagePreview, and the production code returns the unit `FileTarget::ImagePreview` (no layout payload), so there is nothing to assert. Noted only because the comment "representative cross-product of the orthogonal flags" omits the layout dimensions; harmless.

- **The two `editor_choices` `SystemDefault` and `ExternalEditor(Editor::VSCode)` are net-new in the test.** Neither was in the old binary tests. The cross-product correctly demonstrates that ImagePreview wins over all four `EditorChoice` variants; this is a strict superset of the spec's "regardless of `editor_choice`" requirement. Good.
